# DERIVED_METRICS_SPEC

이 문서는 `wed_dashboard` 에서 사용하는 **파생 지표 / 계산값** 기준 초안입니다.  
즉 DB에 그대로 저장하는 값이 아니라, 원천 데이터를 기반으로 계산해서 내려주는 지표를 정리합니다.

## 1. 왜 분리하나

현재 대시보드에는 원천 데이터만으로는 바로 표시할 수 없는 값이 많습니다.

예시:

- OEE
- 목표 달성률
- 시간당 생산량
- 최근 10분 NG
- 불량률
- 전일 대비 / 주간 평균 대비
- 리스크 점수
- 예상 종료 생산량

이 값들은 DB 컬럼을 직접 만드는 것보다

- SQL 집계
- 백엔드 서비스 계산
- 배치/스트림 집계

로 계산하는 편이 자연스럽습니다.

## 3. 계산 예시

### 시간당 생산량

```text
최근 1시간 생산량 합계 / 1시간
```

또는

```text
현재 시점 기준 직전 60분 생산량
```

### 최근 10분 NG

```text
최근 10분 동안 발생한 defect_count 합계
```

### 목표 달성률

```text
(현재 생산량 / 현재 시점 목표 생산량) * 100
```

### 예상 종료 생산

```text
현재 누적 생산량 + (현재 시간당 생산량 × 잔여 작업 시간)
```

### 불량률

```text
(불량 수 / 검사 수) * 100
```

### 전일 대비

```text
현재 불량률 - 전일 동시간대 불량률
```

### 주간 평균 대비

```text
현재 불량률 - 최근 7일 동일 기준 평균
```

### 리스크 점수

예시 규칙:

```text
다운타임 점수 + 불량률 점수 + 미해결 알람 점수 + 온도 경고 점수
```

즉 현재 mock의 `R87`, `R68` 같은 값은 원천 데이터가 아니라 **룰 기반 계산 결과**로 보는 것이 맞습니다.

## 4. DB에 저장할지 여부

파생 지표는 다음 3가지 방식 중 하나로 운용할 수 있습니다.

### A. 조회 시 실시간 계산

장점:

- 실시간성 높음
- 저장 중복 적음

단점:

- 복잡한 집계는 응답이 느릴 수 있음

### B. 집계 테이블 저장

장점:

- 조회 빠름
- 대시보드 안정적

단점:

- 집계 주기 관리 필요

### C. 혼합 방식

권장 방식입니다.

- 원천 데이터는 DB 저장
- 무거운 지표는 집계 테이블
- 가벼운 값은 조회 시 계산

## 5. 현재 mock에서 파생 데이터에 해당하는 필드

아래 값들은 대부분 파생 데이터입니다.

- `kpis[].value` 중 집계성 값
- `lineTemperature` 의 상태 판정
- `hint`
- `riskOverall`
- `riskLines[].riskScore`
- `pendingActions`
- `promoMonthlyCompare`

즉 나중에 백엔드 연결 시 이 값들은 DB 컬럼을 그대로 찾기보다,
**어떤 원천 데이터를 합쳐서 만든 값인지**를 먼저 정의해야 합니다.
