# web_dashboard 더미데이터 흐름 쉬운 설명

이 문서는 `web_dashboard`에서 말하는 "더미데이터"가

- 어디서 만들어지고
- 어디에 쌓이고
- 어떤 경우 화면에 보이고
- 실시간처럼 보이는 값과 기간 비교 값이 어떻게 달라지는지

를 팀원이 빠르게 이해할 수 있게 쉽게 풀어쓴 문서입니다.

---

## 1. 먼저 한 줄 요약

현재 대시보드에는 데이터가 크게 3갈래로 있습니다.

1. 프론트 코드 안에 박혀 있는 `mock/fallback` 데이터
예시: 시간당 생산량 119 pcs 근데 이건 예전에 DB 구현하기 전
2. 서버 코드 안에 박혀 있는 `mock bundle` 데이터
-> server_api/src/modules/dashboard/service.py 파일 안에 파이썬 코드로 직접 적어둔 샘플 응답 묶음  
예시: 최근 10분 NG = 12건, 04:00 actual 2.7
3. DB 테이블에 실제로 쌓인 뒤 서버가 집계해서 만드는 live 데이터

즉, "더미데이터"라고 부르는 것은 사실 한 종류가 아니라,

- 코드 안에 미리 적어둔 샘플값
- DB에 넣어 놓고 실제처럼 집계하는 샘플 레코드

가 섞여 있을 수 있습니다.

---

## 2. 가장 쉬운 그림

```text
[브라우저 web_dashboard]
        |
        | 1) 우선 실제 API 호출
        v
/api/v1/dashboard/web/{tab}
        |
        +--> [서버 service.py]
        |         |
        |         | 2) DB에서 조회 가능하면
        |         v
        |     [repository.py -> DB 집계]
        |         |
        |         v
        |     live 응답 생성
        |
        +--> [서버 내부 mock bundle]
        |         |
        |         v
        |     live가 부족하면 일부 블록 보강 (현재 사용하는 것은 없음)
        |
        v
[정상 응답]

브라우저에서 API 자체가 실패하면
        |
        v
[dashboard_api.js 안의 프론트 mock bundle 사용]
```

핵심은 이겁니다.

- API가 정상이고 DB에도 데이터가 있으면 live 데이터가 우선입니다.
- live가 일부 부족하면 서버가 mock 조각을 섞을 수 있습니다.
- API 호출 자체가 실패하면 프론트가 자기 안의 mock으로 화면을 유지합니다.

---

## 3. "쌓이는" 데이터와 "안 쌓이는" 데이터를 구분하면 쉽다

### A. 안 쌓이는 더미 -> 지금은 DB에서 읽어오는걸로 확인!!!!!!!

이건 코드 파일 안에 숫자와 배열을 직접 써둔 형태입니다.

- 프론트: `dashboard_api.js`의 `getMockBundle(tab)`
- 서버: `service.py`의 `_build_mock_bundles()`

이 데이터는 DB에 insert 되는 구조가 아닙니다.
파일 안에 미리 적어둔 샘플값이라서, 서버를 다시 띄워도 같은 값이 그대로 나옵니다.

예:

- `"오늘 생산량 32528"`
- `"이번 달 생산 187400"`
- `"LINE C 다운"`
- `"최근 10분 NG 12건"`

이런 값이 코드에 박혀 있으면 "쌓인 것"이 아니라 "적어둔 것"입니다.

### B. 실제로 쌓이는 데이터

이건 DB 테이블에 row가 누적되는 형태입니다.

대표 테이블:

- `production_records`
- `equipment_status_history`
- `inspection_results`
- `defect_results`
- `event_logs`

이쪽은 시간이 지나면 row가 늘어날 수 있습니다.
그래서 오늘 누적, 이번 달 누적, 최근 7일 같은 값이 계산 가능합니다.

---

## 4. 프론트 쪽 더미데이터는 어떻게 만들어지나 -> 이건 이제 의미 없음XXXXXXXXXXXXXXX

프론트 파일 `dashboard_api.js` 안에 `getMockBundle(tab)` 함수가 있습니다.

여기서 탭별로:

- `worker`
- `qa`
- `manager`
- `promo`

화면 전체에 필요한 카드, 차트, 리스트 값을 자바스크립트 객체로 직접 만들어 둡니다.

쉽게 말해:

```text
화면이 필요로 하는 완성형 JSON 샘플을
프론트 코드 안에 미리 적어둠
```

이 데이터의 목적은 보통 2가지입니다.

1. 백엔드가 아직 완성되지 않았을 때 화면 개발용
2. API 장애가 나도 최소한 화면이 깨지지 않게 하는 fallback 용도

즉 프론트 mock은 "화면 유지용 비상 샘플"에 가깝습니다.

---

## 5. 서버 쪽 더미데이터는 어떻게 만들어지나

서버 파일 `service.py` 안에는 `_build_mock_bundles()`가 있습니다.

여기서도 화면 단위로:

- KPI
- assistive 정보
- datasets

를 파이썬 객체로 직접 만들어 둡니다.

이 서버 mock의 역할은 단순 샘플 보관을 넘어서,
"live 데이터가 일부만 준비돼도 화면 응답 구조를 유지"하는 데 있습니다.

즉 서버는 이런 식으로 움직입니다.

```text
기본은 mock bundle 복사
-> 그 위에 live 데이터로 덮어쓰기
-> live가 없는 블록은 mock 잔존
-> coverage 정보로 무엇이 live/mock인지 표시
```

그래서 응답이 완전한 live만 있는 것이 아니라,
일부 화면은 한동안 "하이브리드 응답"일 수 있습니다.

---

## 6. live 데이터는 어디에 쌓이고 어떻게 쓰이나

live 데이터는 DB 테이블에 쌓인 row를 서버가 조회해서 만듭니다.

대표 흐름은 아래와 같습니다.

```text
현장/시뮬레이션/적재 프로세스가 DB 테이블에 row 저장
        |
        v
repository.py 가 날짜/라인/교대 기준으로 조회
        |
        v
service.py 가 KPI/차트/리스트 구조로 조립
        |
        v
/api/v1/dashboard/web/{tab} 응답
        |
        v
브라우저 렌더
```

예를 들어:

- `production_records`가 쌓이면
  - 오늘 생산량
  - 이번 달 누적 생산량
  - 최근 7일 생산량
  - 라인별 생산량

을 계산할 수 있습니다.

- `inspection_results`가 쌓이면
  - 검사 수량
  - 합격/불합격 수량
  - 양품률

을 계산할 수 있습니다.

- `defect_results`가 쌓이면
  - 불량 유형 TOP
  - 불량률 추세
  - 최근 NG 목록

을 계산할 수 있습니다.

- `equipment_status_history`가 쌓이면
  - RUN/IDLE/DOWN 상태
  - 가동률/OEE 관련 값
  - 다운타임 이력

을 계산할 수 있습니다.

- `event_logs`가 쌓이면
  - 이벤트 타임라인
  - 최근 공지/로그

를 만들 수 있습니다.

## 7. "실시간처럼 보이는 값"은 실제로 어떻게 계산되나

실시간이라고 해도 websocket으로 초당 밀어주는 구조가 아니라,
현재 시점 기준 snapshot을 다시 계산해서 내려주는 방식에 가깝습니다.

즉 흐름은:

1. 브라우저가 일정 주기로 번들 API 호출
2. 서버가 현재 날짜 기준 `snapshot` 조회
3. DB에 이미 쌓인 row를 합산/집계
4. 그 시점의 결과를 다시 응답

그래서 "실시간형"은 보통 이런 뜻입니다.

- 오늘 0시부터 지금까지 누적
- 현재 시점까지의 최신 상태
- 최근 10분/최근 1시간 집계

즉 "새 row가 DB에 계속 들어오면 실시간처럼 변해 보이는 구조"입니다.

---

## 8. 기간 비교 데이터는 어떻게 만들어지나

기간 비교는 별도로 2~3개월 raw 데이터를 통째로 프론트에 넘기는 개념이라기보다,
서버가 `date_from` ~ `date_to` 범위를 집계해서 비교용 배열로 만드는 방식입니다.

현재 서버 규칙은:

- `date_from`, `date_to`가 둘 다 있으면 기간 비교 모드 가능
- 단, 최대 범위는 31일
- 같은 날짜면 기간 비교가 아니라 단일 날짜 조회

즉 현재 구현 기준으로 일반 period compare는
"최대 한 달짜리 비교 집계"라고 이해하는 것이 맞습니다.

---

## 9. promo 화면의 주간/월간 값은 왜 길게 보이나

`promo` 화면은 일반 카드보다 기간감이 길어 보일 수 있습니다.

이유는 서버 snapshot 안에 기본적으로 아래 값들이 들어가기 때문입니다.

- 오늘 누적 생산량
- 이번 달 누적 생산량
- 최근 7일 생산량

즉 `promo`는:

- `promoWeekProduction`: 최근 7일
- `promo_month_output`: 이번 달 누적

을 같이 보여줍니다.

그래서 사용자가 보기에는 "몇 달치 더미가 깔려 있는 것처럼" 느껴질 수 있지만,
현재 코드 기준 핵심은

- 주간 7일
- 월 누적
- 일반 기간 비교 최대 31일

입니다.

---

## 10. 실제 화면에 값이 보일 때 어떤 경로를 탔는지 판단하는 법

### 1) 프론트 fallback 인지

보통 이런 경우입니다.

- 브라우저에서 `/api/v1/dashboard/web/*` 호출이 실패
- 프론트 `dashboard_api.js`가 자체 mock을 사용

이 경우는 아예 브라우저 쪽 샘플값이 렌더됩니다.

### 2) 서버 mock 인지

API는 성공했는데 값 일부가 샘플처럼 보이면 서버 mock이 섞였을 수 있습니다.

이 경우는 서버가:

- mock bundle을 기본으로 만들고
- live 가능 항목만 덮어쓴 뒤
- 나머지는 mock으로 남겨

응답했을 가능성이 큽니다.

### 3) DB live 인지

이 경우는 repository가 실제 테이블을 읽어서 집계한 값입니다.

특징:

- 날짜를 바꾸면 값이 같이 바뀜
- 최근 7일/월 누적/라인 필터 변화가 반영됨
- DB row가 늘어나면 수치도 바뀜

---

## 11. 헷갈리지 않게 딱 구분하면

### "코드 안 mock"

- 사람이 직접 숫자 적어둔 샘플
- 쌓이지 않음
- 파일 수정 전까지 거의 안 바뀜

### "DB 기반 샘플/더미"

- DB row 형태로 들어간 샘플
- 시간/날짜 기준으로 누적 집계 가능
- 오늘, 최근 7일, 이번 달 같은 계산이 가능

### "live 응답"

- service/repository가 DB 값을 읽어 API 응답으로 조합한 결과
- 화면에는 가장 실제 데이터처럼 보임

---

## 12. 지금 프로젝트 기준 결론

현재 `web_dashboard`에서 "더미데이터 흐름"은 아래처럼 이해하면 가장 정확합니다.

1. 프론트에는 비상용 mock bundle이 있다.
2. 서버에도 기본 mock bundle이 있다.
3. 서버는 가능하면 DB live 데이터를 읽어 mock 위에 덮어쓴다.
4. DB에 row가 쌓이면 오늘/주간/월누적 같은 값이 계산된다.
5. 기간 비교는 가능하지만 현재 일반 범위는 최대 31일이다.
6. 그래서 화면에는 순수 mock, 순수 live, hybrid가 모두 나타날 수 있다.

## 13. 팀에서 이렇게 말하면 가장 덜 헷갈린다

- "프론트 fallback mock"
- "서버 mock bundle"
- "DB 적재 데이터"
- "DB 집계 live 응답"
- "period compare 집계"

반대로 그냥 전부를 묶어서 "더미데이터"라고 부르면,

- 코드 샘플값인지
- DB에 쌓인 샘플 row인지
- live 집계 결과인지

가 섞여서 오해가 생기기 쉽습니다.
