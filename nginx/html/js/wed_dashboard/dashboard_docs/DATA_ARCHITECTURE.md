# wed_dashboard 데이터 구조 개요

이 문서는 `wed_dashboard` 에서 사용하는 데이터를 아래 3개 층으로 나누어 설명합니다.

1. DB에 저장되는 원천 데이터
2. 서버에서 계산/집계되는 파생 데이터
3. 프론트 화면 표시용 응답 데이터

현재 프로젝트 상태 기준:

- 프론트는 이미 존재
- mock 데이터는 `dashboard_api.js` 에 존재
- 공식 DB 설계 문서와 공식 API 스키마 문서는 아직 없음

따라서 이 문서는 **백엔드 연동 전 정리용 초안**입니다.

## 1. 계층 구분

### A. 원천 데이터

센서/설비/검사/알람/생산 실적처럼 실제 현장에서 발생하는 데이터입니다.  
이 데이터는 DB에 저장되는 대상입니다.

예시:

- 설비 상태 이력
- 생산량 실적
- 공정별 검사 결과
- 불량 판정 결과
- 알람 발생/해제 이력
- ACK 처리 이력
- 라인/설비/LOT 기본 정보
- 교대/작업자/근무 시간 정보

### B. 파생 데이터

원천 데이터를 바탕으로 서버 또는 SQL 집계에서 계산되는 값입니다.  
항상 DB 테이블에 저장할 필요는 없고, 조회 시 계산하거나 materialized view/집계 테이블로 운용할 수 있습니다.

예시:

- OEE
- 목표 달성률
- 예상 종료 생산량
- 시간당 생산량
- 최근 10분 NG
- 불량률 추세
- 라인별 리스크 점수
- 재검 우선순위
- 전월 대비 비교 값

### C. 표시용 데이터

프론트 화면에 맞게 조합한 응답 형태입니다.  
DB에 그대로 저장하는 개념이 아니라, API 응답에서 만들어지는 구조입니다.

예시:

- `worker.kpis`
- `qa.topDefects`
- `manager.pendingActions`
- `promoTicker`
- `hint`
- `riskOverall.reason`

## 2. 권장 설계 원칙

권장 기준은 아래와 같습니다.

- DB에는 가능한 한 **원천 데이터** 위주로 저장
- 집계성 지표는 서버/API 계층에서 계산
- 프론트 화면 구조는 DB 구조와 1:1로 맞추지 말고 API에서 조합
- 화면용 문구는 DB 컬럼보다 서비스 계층 조합 결과로 만드는 것이 유리

## 3. 화면별 데이터 분해

### worker

원천 데이터가 필요한 항목:

- 라인별 생산량
- 설비 상태
- 설비별 NG 수
- 라인 온도
- 최근 이벤트

파생 데이터:

- 시간당 생산량
- 현재 라인 생산량
- 최근 10분 NG
- 가동률
- 액션 큐 우선순위

표시용 데이터:

- `hint`
- `actionQueue[].reason`
- `events[].meta`

### qa

원천 데이터가 필요한 항목:

- 검사 결과
- LOT별 불량 결과
- 재검 큐
- 품질 이슈 로그
- 시간대별 불량률 산출용 원천 데이터

파생 데이터:

- 불량률
- 재검 대기 건수
- 검사 현황 요약
- 불량 원인 기여도
- 전일 대비/주간 평균 대비

표시용 데이터:

- `추세 상태`
- `QA 알림`
- `issues[].action`
- `issues[].owner`

### manager

원천 데이터가 필요한 항목:

- 라인별 생산량
- 라인별 설비 상태
- 라인별 불량 결과
- 알람 발생/ACK 상태
- 설비별 다운타임

파생 데이터:

- OEE
- 목표 달성률
- 예상 종료 생산
- 라인별 OEE
- 리스크 점수
- 즉시 조치 Top 3

표시용 데이터:

- `riskOverall.reason`
- `pendingActions[].summary`
- `events[].text`

### promo

원천 데이터가 필요한 항목:

- 일별 생산량
- 라인 상태
- 현재 알람
- 불량 유형 집계
- 월간 비교용 실적

파생 데이터:

- 오늘 생산량
- 이번 달 생산량
- 전체 OEE
- 현재 불량률
- 납기 달성률
- 전월 대비 값

표시용 데이터:

- 하단 티커 문구
- 배지 텍스트
- 카드별 문구 포맷

## 4. 다음 문서와의 관계

이 문서는 전체 구분 문서입니다.

- DB 원천 데이터 기준: `DB_SOURCE_DATA_SPEC.md`
- API 응답 기준: `API_RESPONSE_SCHEMA.md`
- 계산/파생 지표 기준: `DERIVED_METRICS_SPEC.md`

즉,

- DB 설계가 필요하면 `DB_SOURCE_DATA_SPEC.md`
- 프론트 연동이 필요하면 `API_RESPONSE_SCHEMA.md`
- KPI/집계 로직이 필요하면 `DERIVED_METRICS_SPEC.md`

를 먼저 보면 됩니다.
