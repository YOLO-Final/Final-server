# 작업자 화면 수정 설계 기준서

## 1. 문서 목적

본 문서는 기존 `worker` 작업자 화면을 작업자 역할에 맞는 즉시 대응형 화면으로 개편하기 위한 FE/디자인 공통 기준 문서이다.
목적은 단순 시각 개선이 아니라, 정보 우선순위를 재구성하여 작업자가 첫 화면에서 바로 이상 위치와 조치 우선순위를 판단할 수 있도록 화면 구조를 정리하는 데 있다.

본 문서는 아래를 동시에 만족해야 한다.
- 디자이너가 ASCII 와이어프레임을 기준으로 실제 UI 시안을 구체화할 수 있어야 한다.
- FE 개발자가 어떤 블록을 삭제/통합/축소/모달화해야 하는지 바로 이해할 수 있어야 한다.
- 발표/시연 시에도 왜 이런 구조로 바꾸는지 설명 가능한 수준의 논리를 가져야 한다.

- 작업자 화면(`worker`)

## 2. 기존 화면 문제 요약

### 정보 과밀
기존 작업자 화면은 KPI, 상태 그리드, 알람, 이벤트, 미처리 목록, 차트, 로그, 이력 테이블이 동시에 배치되어 있어 첫 화면의 핵심 판단력이 떨어진다.

### 스크롤 발생
한 화면에서 모두 보여주려는 구조 때문에 1920×1080 기준에서도 정보 밀도가 높고, 하단 영역이 길어져 무스크롤 원칙이 훼손된다.

### 알림/조치 정보 중복
다음 3개 영역은 모두 작업자의 주의 환기와 대응 유도를 목적으로 하지만, 각각 분리되어 있어 중복 인지가 발생한다.
- `고심각 알람 Top5`
- `내 근무조 미처리 3건`
- `즉시 조치 Top3`

### 텍스트/표 중심 구조의 한계
실시간 NG 로그, 알람 표, 정지/재가동 이력 등은 읽는 데 시간이 걸리며, 현장 오퍼레이터에게는 첫 화면에서 과한 정보다.

### 작업자 역할과 맞지 않는 분석성 정보 과다
작업자 화면은 분석 보고용이 아니라 `이상 위치 식별 + 초동 대응`이 목적이다.
기존 화면은 일부 분석형 요소가 강해 작업자 역할에 비해 무겁다.

## 3. 작업자 화면 역할 재정의

### 한 줄 목적
지금 멈출 위험 설비와 즉시 조치 대상을 바로 식별하는 화면

### 첫 판단 질문
- 지금 어디가 비정상인가?
- 내가 지금 무엇부터 처리해야 하나?

### 화면 정의
작업자 화면은 `분석형 화면`이 아니라 `즉시 대응형 화면`으로 정의한다.

### 설계 관점 핵심
- 현재 상태를 빨리 읽을 수 있어야 한다.
- 조치 우선순위가 분명해야 한다.
- 상세 로그와 이력은 1차 화면이 아닌 상세 보기로 내려야 한다.
- 중앙 상태 보드와 우측 액션 영역의 역할이 명확히 구분되어야 한다.

## 4. 개편 핵심 원칙

- 무스크롤 1화면을 우선 기준으로 설계한다.
- 중앙 `라인 × 설비 상태 보드`가 화면의 최상위 위계를 가진다.
- 우측은 `무엇부터 처리할지`를 보여주는 액션 중심 패널이어야 한다.
- 하단은 분석 차트가 아니라 즉시 해석 가능한 보조 시각화와 사건 맥락 요약만 둔다.
- 긴 로그/이력/전체 목록은 팝업/모달/오버레이로 분리한다.
- 숫자와 텍스트 나열보다 상태 배지, 비교 바, 상태 셀 등 시각 인지가 빠른 표현을 우선한다.
- 카드/블록 수는 5~7개 수준으로 제한한다.
- 다중 분석 차트, 긴 테이블, 중복 경보 블록은 메인 화면에 두지 않는다.

## 5. 메인 화면 정보 구조 개편안

### 개편 방향 요약
기존 작업자 화면은 `많이 보여주는 화면`에서 `먼저 보여줘야 할 것만 남기는 화면`으로 바꾼다.

### 줄일 것
- 고심각 알람 대형 리스트
- 미처리 대형 리스트
- 실시간 NG 로그 테이블
- 정지/재가동 이력 테이블
- 하단 다중 차트
- 설명성 텍스트 블록

### 통합할 것
- `고심각 알람 Top5`
- `내 근무조 미처리 3건`
- `즉시 조치 Top3`

위 3개를 `우측 즉시 대응 큐`로 통합한다.

### 남길 것
- 상단 KPI 3개
- 중앙 `라인 × 설비 상태 보드`
- 우측 `즉시 대응 큐`
- 하단 `라인별 현재 처리율 / 정체 상태`
- 하단 `최근 이벤트 요약 3건`
- 조건부 `ML 보조 배너`

### 시각화로 대체할 것
기존 하단 알림/미처리 텍스트 영역은 `라인별 현재 처리율 / 정체 상태 비교 바`로 대체한다.

### 최종 정보 구조 표

| 영역명 | 위치 | 1차 노출 정보 | 사용자 질문에 답하는 역할 | 클릭 시 상세 방식 | 비고 |
|---|---|---|---|---|---|
| 상단 KPI 카드 | 상단 | 확인 필요 건수, DOWN 설비 수, 10분 NG | 현재 위험 수준이 어느 정도인가 | 중앙 중형 모달 | 요약형 수치 카드 |
| 라인 × 설비 상태 보드 | 중앙 | 라인/설비 상태, 이상 셀 강조 | 어디가 문제인가 | 우측 슬라이드 패널 | 최우선 메인 영역 |
| 즉시 대응 큐 | 우측 | 우선순위 1~3, 상태 배지, 사유, 액션 버튼 | 무엇부터 처리해야 하나 | 중앙 대형 모달 | 통합 알림/액션 영역 |
| 라인별 현재 처리율 / 정체 상태 | 하단 좌측 | 라인별 처리율, 상태 라벨, 정체 비교 | 어느 라인이 실제로 막혔는가 | 중앙 중형 모달 | 보조 시각화 |
| 최근 이벤트 요약 3건 | 하단 우측 | 최근 사건 3건 요약 | 방금 무슨 일이 있었는가 | 우측 슬라이드 패널 | 맥락 보조 |
| ML 보조 배너 | 하단/보조 | 신뢰도 높은 힌트 1건 | 선제 확인이 필요한가 | 소형 오버레이 | 조건부 노출 |

## 6. 통합 알림 영역 설계안

### 왜 통합하는가
기존 `고심각 알람`, `미처리`, `즉시 조치`는 모두 작업자에게 "지금 봐야 할 일"을 알려주는 정보다.
이들을 각각 큰 독립 블록으로 유지하면 정보가 분산되고 우선순위가 흐려진다.
작업자 화면에서는 여러 경고 리스트보다 하나의 액션 큐가 더 빠르다.

### 통합 영역 정의
영역명:
- `즉시 대응 큐`

표시 단위:
- 최대 3건

각 항목 구성:
- 우선순위 번호
- 대상 라인/설비
- 상태 배지 (`DOWN`, `ALERT`, `UNACK`, `DUE`, `HANDOFF`)
- 한 줄 사유
- 대표 알람ID 또는 마감/경과 시간
- 1차 액션 버튼 (`ACK`, `확인`, `인계`, `상세`)

### 정보 흡수 방식
- `고심각 알람 Top5` → 대표 알람ID, 심각도, 발생 시각만 흡수
- `내 근무조 미처리 3건` → 마감 시각, 담당 상태, 인계 상태만 흡수
- `즉시 조치 Top3` → 큐의 메인 구조로 유지

### 메인 화면에서 제거하고 상세로 내릴 정보
- 알람 전체 목록
- 미처리 전체 목록
- 조치 메모/인계 이력
- 설비 이벤트 로그
- 정지/재가동 이력
- 연관 알람 전체 목록
- 작업 이력 전체

## 7. 대체 시각화 설계안

### 최종 선택
- `라인별 현재 처리율 / 정체 상태 비교 바`

### 선택 이유
이 시각화는 작업자에게 필요한 `현재 정체 라인 식별`에 가장 직접적으로 대응한다.
시간대별 생산량 차트처럼 흐름 분석을 요구하지 않고, 지금 어느 라인이 실제로 막혀 있는지를 바로 보여준다.

### 무엇을 보여주는가
- LINE A / B / C / D
- 막대 길이 = 현재 처리율
- 상태 라벨 = 정상 / 주의 / 정체 / NG 영향
- 필요 시 상태 색 구분

### 중앙 상태 보드와의 관계
- 중앙 상태 보드: 설비 단위의 위치 식별
- 하단 비교 바: 라인 단위의 흐름 저하 식별

즉, 두 영역은 중복이 아니라 보완 관계다.

### 다른 대안보다 더 적합한 이유
`시간대별 생산량 바차트 + 목표선`은 작업자 화면에서는 분석성 해석을 유도할 가능성이 크다.
`DOWN/ALERT 설비 분포 히트존`은 상태 보드와 의미 중복이 생길 수 있다.
반면 `라인별 현재 처리율 / 정체 상태 비교 바`는 현재 행동 연결성이 가장 높다.

## 8. 상세 보기(모달/슬라이드/오버레이) 설계 원칙

### 메인 영역별 상세 연결 방식

| 메인 영역 | 상세 방식 | 목적 |
|---|---|---|
| 상단 KPI 카드 | 중앙 중형 모달 | 수치 기준, 최근 변화, 영향 대상 확인 |
| 상태 보드 셀 | 우측 슬라이드 패널 | 설비 상태 및 즉시 조치 확인 |
| 즉시 대응 큐 | 중앙 대형 모달 | 조치 절차, 인계, 작업 실행 |
| 처리율/정체 상태 | 중앙 중형 모달 | 라인 병목/흐름 저하 확인 |
| 최근 이벤트 요약 | 우측 슬라이드 패널 | 사건 상세 및 전후 맥락 확인 |
| ML 보조 배너 | 소형 오버레이 | 힌트 근거 및 추천 확인 포인트 확인 |

### 상단 고정 정보 원칙
모든 상세 팝업은 상단에 아래를 고정한다.
- 제목
- 대상(라인/설비/이벤트/KPI)
- 현재 상태
- 경과 시간 또는 마감 시각
- 대표 알람ID 또는 핵심 식별자
- 핵심 액션 버튼
- 닫기/뒤로

### 클릭 확장/탭/2차 상세 분리 원칙
- 설명성 정보는 `클릭 확장(아코디언/더보기)` 처리
- 구조 비교가 필요한 정보는 `탭 2개 수준`으로 분리
- 로그/이력/원문/메모 전체는 `2차 상세`로 분리
- 메인 팝업 본문은 핵심만 유지하고 길어지지 않게 한다

### 스크롤 최소화 원칙
- 전체 팝업 세로 스크롤은 지양한다.
- 필요한 경우 특정 `로그 슬롯` 영역에만 내부 스크롤을 허용한다.
- 내부 스크롤 영역도 행 수 제한을 둔다.

### 절대 메인 팝업에 길게 두지 말아야 할 것
- 원문 로그 전체
- 정지/재가동 이력 전체
- 작업 이력 전체
- 연관 알람 전체 목록
- 장문 메모 전체
- 계산 로직 상세 설명
- 다중 표/다중 차트

## 10. 대표 상세 팝업 구조 예시

### 10-1. 설비 상세 슬라이드 패널 예시


## 12. FE 반영 관점 수정 포인트

### 삭제할 것
- 고심각 알람 Top5 독립 블록
- 내 근무조 미처리 3건 독립 블록
- 실시간 NG 로그 대형 테이블
- 정지/재가동 이력 테이블
- 큰 알람 표
- 분석형 다중 차트 병렬 노출

### 통합할 것
- `고심각 알람 Top5`
- `내 근무조 미처리 3건`
- `즉시 조치 Top3`

위 3개를 `즉시 대응 큐`로 통합

### 축소할 것
- 이벤트 정보는 요약 3건만 메인에 노출
- ML 정보는 조건부 1건 배너만 노출
- KPI는 3개까지만 유지

### 이동할 것
- 로그/이력/메모/연관 알람 → 상세 팝업 및 2차 상세로 이동
- 설명성 정보 → 아코디언/탭/더보기로 이동

### 새로 시각화로 대체할 것
- 하단 알림/미처리 텍스트 영역 → 라인별 현재 처리율 / 정체 상태 비교 바

### 구현 우선순위 제안
1. 메인 레이아웃 구조 재정렬
2. 우측 즉시 대응 큐 통합 구현
3. 하단 비교 바 시각화 반영
4. 상태 보드 상세 슬라이드 패널 정리
5. 즉시 대응 상세 모달 정리
6. KPI/이벤트/ML 상세 연결
7. 2차 상세 흐름 및 내부 스크롤 슬롯 정리

### FE 구현 시 주의사항
- 상태 보드가 화면 내 최상위 시각 위계를 가져야 한다.
- 즉시 대응 큐는 카드 리스트가 아니라 액션 패널처럼 작동해야 한다.
- 메인 화면에 긴 목록을 다시 올리지 않는다.
- 전체 팝업 세로 스크롤이 기본 동작이 되지 않도록 구조를 잡는다.
- 내부 로그 슬롯만 제한적으로 스크롤되도록 설계한다.

## 13. 최종 기대 효과

### 작업자 시선 흐름 개선
상단 KPI → 중앙 상태 보드 → 우측 즉시 대응 큐 → 하단 처리율/이벤트 순으로 읽히는 명확한 흐름을 확보한다.

### 즉시 대응성 향상
작업자가 첫 화면에서 이상 위치와 우선 조치 대상을 즉시 식별하고, 바로 액션 버튼 또는 상세 팝업으로 이동할 수 있다.

### 정보 중복 제거
고심각 알람, 미처리, 즉시 조치 정보를 통합하여 중복 알림 구조를 제거한다.

### 시연 설명 용이성
"어디가 문제인지"와 "무엇부터 할지"가 명확한 구조이므로 발표/시연 시 설명이 쉬워진다.

### 확장 가능한 상세 구조 확보
메인 화면은 가볍게 유지하면서도, 상세 팝업과 2차 상세를 통해 필요한 정보 확장이 가능해진다.
