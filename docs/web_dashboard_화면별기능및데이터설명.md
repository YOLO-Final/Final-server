# 웹 대시보드 화면별 기능 및 데이터 설명서

## 1. 이 문서의 목적
- 오늘 시연에서 `web_dashboard.html` 기준으로, 각 화면이 무엇을 보여주는지 빠르게 설명하기 위한 문서다.
- 발표자는 이 문서를 5~10분 정도 읽으면, 각 탭의 목적, 연결 데이터, 시연 포인트를 바로 말할 수 있다.
- 설명 기준은 실제 화면 구조와 연결 JS/API 코드다. 따라서 단순 소개가 아니라 "무슨 데이터를 어떤 의미로 보여주는가"까지 정리했다.

### 상태 표기 기준
- `실제 확인됨`: 코드에서 직접 확인한 내용
- `코드상 추정`: 이름, 구조, 처리 흐름으로 보아 그렇게 해석하는 것이 합리적인 내용
- `추가 확인 필요`: 코드만으로는 운영 기준을 단정하기 어려운 내용

---

## 2. 전체 화면 구성 한눈에 보기

### 현재 존재하는 화면/탭 목록
| 화면 | 역할 | 한 줄 목적 | 상태 |
|---|---|---|---|
| 작업자 | 현장 작업자 | 지금 라인과 설비에서 어떤 이상이 생겼는지 빠르게 확인 | `실제 확인됨` |
| 품질(QA) | 품질 담당자 | 불량률, 주요 불량 원인, 재검 대상을 추적 | `실제 확인됨` |
| 관리자 | 운영/생산 관리자 | 생산성과 리스크를 한 번에 보고 우선 조치 대상을 정함 | `실제 확인됨` |
| 공용 송출 | 공용 모니터/전광판 | 누구나 한눈에 볼 수 있는 전체 현황 요약 화면 | `실제 확인됨` |

### 공통으로 보이는 요소
- 상단 헤더
  - 사용자/권한, 기준 시각, 마지막 갱신 시각, 데이터 지연 여부 표시
- 필터 바
  - 공장, 라인, 근무조, 기간 선택
- 역할 탭
  - 작업자 / 품질(QA) / 관리자 / 공용 송출 전환
- 알림 버튼
  - 현재 `activeAlarms` 기반 알람/이벤트 목록 팝업 표시
- 상세 보기 패널
  - 카드, 표, 상태 그리드, 링크를 누르면 우측 상세 패널이 열림
- 테마 변경
  - 라이트/다크 선택 가능

### 발표 전에 알아둘 공통 한마디
- 이 대시보드는 "역할별로 필요한 정보만 다르게 보여주는 운영 대시보드"다.
- 같은 데이터라도 작업자는 즉시 조치 중심, QA는 불량 추적 중심, 관리자는 운영 판단 중심으로 구성되어 있다.

## 3. 화면별 기능 설명

### 3-1. 작업자 화면

- 화면 목적
  - 현장 작업자가 지금 바로 확인해야 할 이상 상황과 조치 대상을 빠르게 파악하는 화면
  - `실제 확인됨`

- 화면에서 보이는 주요 구성요소
  - KPI 4개
    - 최근 10분 NG
    - 누적 생산량
    - 목표 달성률
    - 현재 확인 필요 건수
  - ML 이상 징후 카드
  - 라인 x 설비 상태 그리드
  - 최근 알람/이벤트 타임라인
  - Active 알람 표
  - 최근 10분 NG 추세 차트
  - 최근 불량 TOP3 차트
  - 내 근무조 미처리 항목
  - 실시간 NG 로그
  - 최근 정지/재가동 이력
  - `실제 확인됨`

- 제공 기능
  - 설비 상태 칸을 눌러 설비/라인 상세 확인
  - 알람 ID를 눌러 알람 상세 확인
  - 타임라인 항목을 눌러 이벤트 상세 확인
  - 미처리 항목 버튼을 눌러 알람 또는 라인 상세 이동
  - 상단 "확인 필요 건 상세 / 라인 상태 점검 / 이벤트 상세" 빠른 진입 버튼 제공
  - `실제 확인됨`

- 사용 데이터
  - KPI
    - `worker_recent_10m_ng`, `worker_total_produced`, `worker_achievement`, `worker_attention_count`
  - 데이터 블록
    - `statusGrid`, `activeAlarms`, `eventTimeline`, `ngTrend10m`, `topDefects`, `ngLogs`, `downtimeHistory`, `pendingActions`, `handoffLogs`
  - 데이터 출처 구분
    - 실데이터 가능: `vision_result` 기반 live snapshot에서 일부 채움
    - 혼합 가능: live가 없는 블록은 mock 유지
    - 완전 미연동 시: `MOCK_SCREENS.worker` 사용
  - 세부 해석
    - `최근 10분 NG`: 최근 단위 시간 내 불량 개수
    - `상태 그리드`: 라인별 설비 상태(RUN/IDLE/DOWN/MAINT)
    - `실시간 NG 로그`: 최근 검출된 불량 기록
    - `ML 이상 징후`: 예측/보조 성격의 안내 카드
  - `실제 확인됨`

- 시연 멘트 예시
  - "작업자 화면은 지금 현장에서 바로 봐야 하는 정보만 모아둔 화면입니다."
  - "왼쪽 위 KPI로 현재 이상 정도를 먼저 보고, 아래 상태 그리드와 알람 표로 어디 설비에서 문제가 났는지 바로 확인할 수 있습니다."
  - "최근 10분 NG 추세를 같이 보여주기 때문에 순간적인 문제인지, 계속 악화되는지까지 빠르게 판단할 수 있습니다."

- 주의사항
  - `현재 확인 필요 건수`는 live가 없으면 "계약 대기" 또는 mock 값일 수 있다.
  - ML 이상 징후 카드는 실제 측정값이라기보다 보조 판단 카드다. `simulated` 또는 보조성 데이터로 표시될 수 있다.
  - `handoffLogs`는 데이터 구조에는 있지만 작업자 메인 화면 본문에는 직접 크게 노출되지 않고 상세/연관 조회에 더 가깝다.
  - `실제 확인됨`

### 3-2. 품질(QA) 화면

- 화면 목적
  - 불량률 변화, 주요 불량 원인, 재검 우선순위를 품질 담당자가 빠르게 파악하는 화면
  - `실제 확인됨`

- 화면에서 보이는 주요 구성요소
  - KPI 3개
    - 현재 불량률
    - 재검 필요 수량
    - 검사 현황 요약
  - 품질 경보 카드
  - 불량률 추세 차트
  - 주요 불량 기여도 / 원인코드 차트
  - 재검 큐 표
  - NG 상세 목록 / 추적 링크 표
  - `실제 확인됨`

- 제공 기능
  - 원인코드 버튼을 눌러 불량 상세 drilldown
  - 재검 큐의 Inspection, Lot 클릭 시 상세 확인
  - NG 상세 목록의 Lot/설비/시간 링크 클릭 시 추적 상세 확인
  - 상단 "재검 큐 진입 / 판정 근거 보기 / Lot 추적" 빠른 진입 버튼 제공
  - `실제 확인됨`

- 사용 데이터
  - KPI
    - `qa_defect_rate`, `qa_recheck`, `qa_summary`
  - 보조 카드
    - `qa_alert_status`, `qa_pattern_hint`
  - 데이터 블록
    - `defectTrend`, `topDefects`, `ngRows`, `recheckQueue`, `handoffLogs`, `summaryLine`
  - 데이터 출처 구분
    - live 가능: 불량률, 재검 건수, 검사 요약, 최근 NG 목록, 상위 불량 유형
    - 일부는 예측/보조 표현 가능: `qa_pattern_hint`
    - 부족한 블록은 mock으로 유지
  - 세부 해석
    - `불량률 추세`: 시간대별 실제 불량률과 예측값, 그리고 주의/위험 기준선
    - `주요 불량 기여도`: 어떤 불량 유형이 전체 불량에 가장 크게 기여하는지
    - `재검 큐`: 재확인 우선 대상 목록
    - `NG 상세 목록`: 어떤 Lot, 어떤 설비에서, 어떤 불량이 발생했는지 추적용 리스트
  - `실제 확인됨`

- 시연 멘트 예시
  - "QA 화면은 단순히 불량 건수만 보는 화면이 아니라, 왜 불량이 늘었는지와 어떤 건을 먼저 다시 봐야 하는지까지 연결해서 보여줍니다."
  - "불량률 추세와 원인코드를 같이 보면 현재 품질 이슈가 일시적인지, 특정 유형으로 집중되는지 설명하기 좋습니다."
  - "재검 큐와 Lot 추적 링크 덕분에 실제 후속 확인 흐름까지 이어진다는 점이 포인트입니다."

- 주의사항
  - 예측선은 있는 경우에만 의미가 있다. live 데이터에서는 `predicted`가 비어 있을 수 있다.
  - `qa_pattern_hint`는 실제 확정 판정보다 "패턴 탐지 힌트"에 가깝다.
  - `summaryLine`은 화면 본문에 크게 노출되기보다 데이터 보조 설명으로 쓰인다.
  - `실제 확인됨`

### 3-3. 관리자 화면

- 화면 목적
  - 생산 실적, 운영 리스크, 손실 원인을 한 화면에서 보고 우선 조치 대상을 정하는 화면
  - `실제 확인됨`

- 화면에서 보이는 주요 구성요소
  - KPI 3개
    - 전체 라인 OEE
    - 목표 달성률
    - 예상 종료 생산량
  - 운영 리스크 수준 카드
  - 라인 상태 그리드
  - 위험 라인 우선순위 리스트
  - 생산 / 불량 추세 차트
  - 계획 대비 편차 차트
  - 다운타임 Pareto 표
  - 미해결 알람 표
  - 조치 지연 / 인계 이슈
  - 권고 문구 요약
  - `실제 확인됨`

- 제공 기능
  - 리스크 상세, 병목 상세, 권고 상세, 손실 원인 상세, 이벤트 상세로 바로 이동
  - 라인 상태 그리드나 위험 라인 목록을 눌러 라인 상세 확인
  - 다운타임 Pareto 원인코드를 눌러 원인별 상세 확인
  - `실제 확인됨`

- 사용 데이터
  - KPI
    - `mgr_oee`, `mgr_achievement`, `mgr_expected_output`
  - 보조 카드
    - `mgr_risk`
  - 데이터 블록
    - `statusGrid`, `activeAlarms`, `hourly`, `lineCompare`, `statusDistribution`, `downtimePareto`, `riskLines`, `pendingActions`, `handoffLogs`, `recommendation`
  - 데이터 출처 구분
    - live 가능: `hourly`, `recommendation`, 일부 KPI
    - mock 유지 가능: 상태 분포, 라인 비교, 다운타임 Pareto, 위험 라인 등 일부 운영 블록
    - 하이브리드 응답일 가능성 높음
  - 세부 해석
    - `생산 / 불량 추세`: 생산량과 불량률을 함께 보여줘 생산성-품질 관계를 같이 판단
    - `계획 대비 편차`: 라인별 실적이 계획보다 얼마나 앞서거나 뒤처지는지
    - `다운타임 Pareto`: 손실 원인을 큰 순서대로 보여줘 우선 개선 대상을 찾음
    - `권고`: 지금 우선 무엇을 조치해야 하는지 한 줄로 요약
  - `실제 확인됨`

- 시연 멘트 예시
  - "관리자 화면은 현황 모니터링을 넘어서, 어디를 먼저 조치해야 하는지 판단하는 의사결정 화면입니다."
  - "생산량, 불량률, 계획 편차, 다운타임 원인을 같이 보기 때문에 단순 상태판보다 훨씬 운영 판단에 가깝습니다."
  - "특히 위험 라인과 권고 문구는 오늘 당장 우선순위를 설명할 때 유용합니다."

- 주의사항
  - `운영 리스크 수준`은 보조/시뮬레이션 성격일 수 있다.
  - `mgr_oee`는 이름은 OEE지만 현재 코드상 live에서는 실제로 양품률 계열 값으로 대체될 수 있어, 엄밀한 설비 OEE라고 단정하면 안 된다.
  - 차트 렌더 함수에 `statusDistribution` 도넛 차트가 있으나 현재 관리자 레이아웃에는 해당 캔버스가 없어 실제 화면에는 안 보일 가능성이 높다.
  - `실제 확인됨`

### 3-4. 공용 송출 화면

- 화면 목적
  - 공장 전체 현황을 누구나 쉽게 이해할 수 있게 보여주는 요약형 화면
  - `실제 확인됨`

- 화면에서 보이는 주요 구성요소
  - 큰 KPI 3개
    - 오늘 총 생산량
    - 목표 달성률
    - 당일 양품률
  - 현재 시각 표시
  - 롤링 메시지형 주의 포인트
  - 라인 상태 보드
  - 금일 TOP 이슈 3
  - 시간대별 생산 추세 차트
  - TOP3 DEFECT 차트
  - `실제 확인됨`

- 제공 기능
  - 운영 화면으로 돌아가기 버튼
  - 상세 drilldown은 막혀 있음
  - `실제 확인됨`

- 사용 데이터
  - KPI
    - `promo_total`, `promo_achievement`, `promo_yield`
  - 데이터 블록
    - `statusGrid`, `lineTrend`, `top3`, `topIssues`, `rollingMessage`
  - 데이터 출처 구분
    - 일부 live 가능: 총 생산량, 양품률, 생산 추세, 상위 불량
    - 일부는 mock 유지 가능: 상태 보드, 이슈 카드
  - 세부 해석
    - `롤링 메시지`: 지금 가장 강조해서 보여주고 싶은 주의 포인트
    - `TOP 이슈 3`: 비개발자도 이해하기 쉬운 사건 요약
  - `실제 확인됨`

- 시연 멘트 예시
  - "공용 송출 화면은 현장이나 회의실 모니터에 띄워두는 전체 현황판입니다."
  - "세부 조작보다는 한눈에 상황을 이해하는 데 목적이 있고, 누가 봐도 지금 생산과 품질이 어떤 상태인지 바로 파악할 수 있습니다."
  - "운영 화면은 분석용, 공용 송출은 공유용이라는 차이를 같이 설명하면 이해가 쉽습니다."

- 주의사항
  - 이 화면은 상세 보기 API를 지원하지 않는다.

## 4. 데이터 구조 쉽게 이해하기

### KPI 데이터
- 가장 위에서 숫자로 크게 보여주는 핵심 지표다.
- 예시
  - 작업자: 최근 10분 NG, 누적 생산량
  - QA: 현재 불량률, 재검 필요 수량
  - 관리자: 전체 라인 OEE, 예상 종료 생산량
  - 공용 송출: 오늘 총 생산량, 양품률
- 쉽게 말하면
  - "지금 이 화면에서 제일 먼저 봐야 하는 숫자"
- 포함 정보
  - 값, 단위, 데이터 출처 유형, 클릭 가능 여부
- `실제 확인됨`

### 차트/리스트용 dataset 데이터
- 차트, 표, 리스트, 상태 보드에 들어가는 본문 데이터다.
- 예시
  - `statusGrid`: 라인/설비 상태
  - `activeAlarms`: 현재 알람 목록
  - `defectTrend`: 시간대별 불량률
  - `recheckQueue`: 재검 대상 목록
  - `lineCompare`: 계획 대비 실적 비교
- 쉽게 말하면
  - "숫자 한 개가 아니라 흐름과 맥락을 보여주는 데이터"
- `실제 확인됨`

### 상세보기(detail) 데이터
- 사용자가 항목을 클릭했을 때 보여주는 심화 정보다.
- 구조
  - `summary`: 핵심 속성 요약
  - `logs`: 관련 이력
  - `relatedItems`: 같이 봐야 할 연관 항목
  - `actions`: 조치 기록
- 쉽게 말하면
  - "이게 왜 문제인지, 무엇과 연결되는지 더 깊게 보는 데이터"
- `실제 확인됨`

### sourceType 쉽게 해석하기
| 값 | 쉬운 설명 | 발표 시 표현 |
|---|---|---|
| `actual` | 실제 집계/실측 데이터 | "실제 집계값입니다" |
| `derived` | 계산해서 만든 값 | "실측을 바탕으로 계산한 지표입니다" |
| `simulated` | 가정/시뮬레이션성 값 | "보조 판단용 값입니다" |
| `predicted` | 예측값 | "앞으로의 추세를 보는 참고값입니다" |
| `recommended` | 추천/권고 값 | "시스템이 권고하는 우선 조치입니다" |

### dataMode 쉽게 해석하기
| 값 | 쉬운 설명 |
|---|---|
| `live` | 실제 연동 데이터를 포함한 응답 |
| `mock` | 샘플/예시 데이터 중심 응답 |

### 이 화면의 데이터 운용 방식 한 줄 정리
- 현재 구조는 "실데이터가 있으면 덮어쓰고, 부족한 부분은 mock으로 채우는 하이브리드 방식"이다.
- `실제 확인됨`

## 5. 시연 추천 흐름

1. 관리자 화면부터 시작
- 전체 운영 상황을 한 번에 설명하기 가장 좋다.
- "이 대시보드는 생산, 품질, 리스크를 역할별로 보여준다"는 큰 그림을 먼저 잡기 좋다.

2. 작업자 화면으로 이동
- 관리자 화면에서 보인 이슈가 현장에서는 어떻게 보이는지 연결해서 설명하기 좋다.
- 상태 그리드, 알람, NG 추세를 보여주면 직관적이다.

3. QA 화면으로 이동
- 같은 문제를 품질 관점에서는 어떻게 추적하는지 보여준다.
- 불량률 추세, 원인코드, 재검 큐를 묶어서 설명하면 자연스럽다.

4. 공용 송출 화면으로 마무리
- "운영용 화면"과 "공유용 화면"의 차이를 보여주며 정리하기 좋다.
- 회의실/현장 모니터용 전체 현황판이라는 메시지가 분명하다.

### 발표자가 놓치지 말아야 할 핵심 포인트
- 같은 공장 데이터를 역할별로 다르게 요약해 보여준다는 점
- 숫자만이 아니라 알람, 이력, 상세 추적까지 이어진다는 점
- 실데이터와 mock이 혼합될 수 있으므로, 현재는 시연용 하이브리드 구조라는 점

## 6. 시연 시 주의할 점

### mock / fallback / 미연동 가능성
- 실데이터가 없으면 전체 화면이 mock으로 표시될 수 있다.
- 실데이터가 일부만 있으면 나머지는 mock으로 채워지는 하이브리드 응답이다.
- 화면 상단의 coverage 안내가 있으면 실데이터와 mock 혼합 여부를 직접 보여줄 수 있다.
- `실제 확인됨`

### 실제 운영 데이터처럼 과장하면 안 되는 부분
- 로그인/권한은 현재 서버 인증이 아니라 브라우저 저장소 기반 프론트 제어에 가깝다.
- 필터 UI는 존재하지만, 백엔드 서비스 코드에서는 현재 필터 값을 실제 집계에 거의 반영하지 않는 구조로 보인다.
- ML/예측/리스크/권고 카드는 운영 의사결정 보조 정보로 설명해야 한다.
- `실제 확인됨`

### 질문 받았을 때 안전한 표현
- "현재는 실데이터와 샘플 데이터가 혼합될 수 있는 시연 구조입니다."
- "실시간 집계가 들어오는 블록은 실제 응답을 사용하고, 아직 미연동인 블록은 fallback 데이터로 유지합니다."
- "예측/리스크 카드는 확정 판정이 아니라 보조 판단용 정보입니다."
- "필터와 권한은 화면 시연 기준으로 구현돼 있고, 운영 수준의 인증/권한 체계는 추가 설계 대상입니다."

### 추가로 조심할 부분
- 관리자 화면의 상태 분포 도넛 차트는 코드에는 있지만 현재 실제 레이아웃에는 보이지 않는다.
- `mgr_oee`는 이름상 OEE지만 현재 live 대체 방식상 엄밀한 설비 종합효율 지표라고 단정하지 않는 것이 안전하다.
- 공용 송출 화면은 drilldown 없는 요약판으로 설명하는 것이 맞다.
- `실제 확인됨`
