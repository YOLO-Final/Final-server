## 1. 개편 배경과 문제 정의

### 기존 관리자 화면의 문제
기존 관리자 화면은 KPI, 리스크, 상태 보드, 위험 라인, 차트, Pareto, 알람, 인계 이슈, 권고가 동시에 경쟁하는 구조였다.  
그 결과 첫 화면에서 가장 중요한 질문인 `오늘 목표 리스크는 무엇이고, 어느 라인에 자원을 먼저 넣어야 하나?`에 바로 답하기 어려웠다.

### 왜 텍스트/표 중심 구조를 바꿔야 하는지
- 대형 표와 상세 목록은 상황 판단보다 읽기 부담을 먼저 만든다.
- 차트가 많아질수록 관리자 화면이 `설명용 보고서`처럼 보이고, 우선순위 판단은 늦어진다.
- 권고와 근거가 같은 화면에서 섞이면 화면 위계가 무너진다.
- 메인 화면은 판단용, 모달은 근거 설명용으로 분리해야 의사결정 구조가 선명해진다.

### 관리자 역할에 맞는 화면이 무엇인지
관리자 화면은 현장 설비를 직접 조작하는 작업자 화면이 아니라,  
`성과 확인 + 운영 리스크 파악 + 우선조치 대상 결정`을 5초 안에 끝내는 상황판형 화면이어야 한다.

## 2. 개편 목표와 설계 원칙

### 관리자 화면의 핵심 질문
- 오늘 목표 리스크는 무엇인가?
- 어느 라인에 자원을 먼저 넣어야 하는가?

### 첫 화면 목표
- 한 줄로 현재 공장 상황을 요약 가능해야 한다.
- 우선 관리해야 할 라인을 바로 말할 수 있어야 한다.
- 어떤 조치를 먼저 해야 하는지 즉시 결정 가능해야 한다.

### 정보 우선순위 원칙
1. 성과 KPI는 상단에서 현재 상태만 빠르게 확인
2. 운영 리스크는 현재 전체 공장 위험 수준을 정의
3. 즉시 조치 우선순위는 자원 배분 결정을 유도
4. 라인별 위험 우선순위 맵은 집중 대상 라인을 결정
5. 상태·알람 요약은 보조 확인용으로만 유지

### 무스크롤 원칙
- 첫 화면은 무조건 한 화면 무스크롤 구조
- 블록 수는 5개 수준으로 제한
- 목록은 최대 4~5행
- 대형 상세표 금지
- 대형 차트 금지
- 세부 해석은 클릭 후 모달로 분리

### 시연 전달 원칙
- 메인 화면에서 전체 상황을 한 문장으로 설명
- 리스크 상황판으로 핵심 위험 설명
- 위험 라인 맵으로 집중 라인 선택
- 상세 모달에서 근거 제시
- 즉시 조치 상세로 결론 정리

## 3. 첫 화면 정보 구조 확정안

### 첫 화면에 남길 블록
- `성과 KPI 3개`
  - OEE
  - 목표 달성률
  - 예상 생산
- `운영 리스크 상황판`
  - 현재 리스크 수준
  - 주요 원인 2~3개
  - 영향 라인
- `즉시 조치 우선순위`
  - 오늘 가장 먼저 결정해야 할 3건
- `라인별 위험 우선순위 맵`
  - 라인 단위 Risk Score
  - 상태칩
  - 한 줄 영향 요약
- `상태·알람 요약`
  - 공장 상태 분포
  - 고위험 알람 요약

### 첫 화면에서 제거하고 모달로 내릴 블록
- KPI 추세 전체
- KPI 변동 원인 세부 설명
- 리스크 구성 근거 상세
- 원인 비중 원본 데이터
- 라인별 편차 상세
- 다운타임/품질/병목 세부
- 알람 전체 목록
- 설비 상태 로그 전체
- 정비/정지 이력 전체
- 권고 근거/실행 조건 세부

### 블록 우선순위
1. `운영 리스크 상황판`
2. `라인별 위험 우선순위 맵`
3. `즉시 조치 우선순위`
4. `성과 KPI 3개`
5. `상태·알람 요약`

### 시선 흐름
- 상단: 성과 KPI
- 좌중앙: 운영 리스크 상황판
- 우중앙: 즉시 조치 우선순위
- 하단 메인: 라인별 위험 우선순위 맵
- 하단 보조: 상태·알람 요약

### 블록 간 관계
- KPI는 결과 상태를 보여준다.
- 리스크 상황판은 현재 위험의 성격을 요약한다.
- 라인별 위험 맵은 위험 집중 위치를 보여준다.
- 즉시 조치 우선순위는 실행 결정을 만든다.
- 상태·알람 요약은 보조적 운영 문맥을 제공한다.

## 4. 상세 모달 체계 확정안

### 블록-모달 연결 표

| 메인 블록 | 연결 모달 | 메인 블록 책임 | 모달 책임 |
|---|---|---|---|
| 성과 KPI 3개 | `성과 KPI 상세 모달` | 현재 성과 상태 전달 | 추세, 편차, 변동 원인 설명 |
| 운영 리스크 상황판 | `운영 리스크 상세 모달` | 전체 리스크 수준 전달 | 리스크 구성 근거와 영향 설명 |
| 즉시 조치 우선순위 | `즉시 조치 상세 모달` | 자원 배분 우선순위 전달 | 기대 효과, 실행 조건, 관련 라인 정리 |
| 라인별 위험 우선순위 맵 | `라인별 위험 상세 모달` | 집중 라인 식별 | 라인 단위 병목/품질/다운타임 근거 설명 |
| 알람 요약 | `알람 상세 모달` | 현재 알람 수준 요약 | 심각도·라인·조치상태 기준 상세 탐색 |
| 상태 보드 요약 | `상태 보드 상세 모달` | 공장 상태 분포 전달 | 라인→설비 단위 drilldown 확인 |

### 첫 화면과 상세 모달의 책임 분리
- 첫 화면: `무엇이 문제인가`와 `어디를 먼저 봐야 하는가`
- 상세 모달: `왜 그렇게 판단하는가`와 `무엇을 실행해야 하는가`

이 책임 분리가 명확해야 메인 화면 과밀이 방지되고, 시연 시에도 클릭 흐름이 논리적으로 이어진다.

## 5. 모달 인터랙션 최종 결정안

### 스크롤 / 아코디언 / 탭 결정 원칙

| 방식 | 적용 조건 | 관리자 상세 모달에서 적합한 정보 |
|---|---|---|
| 스크롤 | 시간순/목록형, 항목 수 가변, 연속 탐색 필요 | 알람 목록, 이벤트 로그, 설비 상태 로그 |
| 아코디언 | 설명성 정보, 실행 조건, 항상 먼저 볼 필요 없음 | 권고 근거, 실행 전 확인사항, 세부 설명 |
| 탭 | 같은 질문을 다른 관점으로 전환해서 봐야 함 | 추세/편차/원인, 상태/병목/품질, 라인/설비 |

### 6개 모달 각각의 최종 인터랙션 구조

| 모달명 | 주방식 | 보조방식 | 구조 결정 |
|---|---|---|---|
| 성과 KPI 상세 모달 | 탭 | 없음 | `추세 / 편차 / 변동 원인` 탭 |
| 운영 리스크 상세 모달 | 탭 | 아코디언 | `원인 구성 / 영향 라인 / 근거 데이터` 탭 + 하단 근거 접기 |
| 즉시 조치 상세 모달 | 아코디언 | 없음 | 요약 고정 + 실행 전 확인사항 펼침 |
| 라인별 위험 상세 모달 | 탭 | 스크롤 | `상태 / 병목 / 품질 영향` 탭 + 하단 이벤트 로그 스크롤 |
| 알람 상세 모달 | 스크롤 | 없음 | 필터 고정 + 목록 스크롤 |
| 상태 보드 상세 모달 | 탭 | 스크롤 | 라인 전환 탭 + 설비 이벤트 로그 스크롤 |

### 피해야 할 구조
- `탭 + 아코디언 + 스크롤` 3중 중첩
- 모달 전체 세로 스크롤
- 클릭한 블록과 직접 관계 없는 정보 확장
- 모달 안에서 또 대형 표/복수 차트 배치
- 상단 고정 영역보다 하단 상세가 더 강해 보이는 구조

### 중첩 금지 규칙
- 한 모달은 `주방식 1개 + 보조방식 1개`까지만 허용
- 탭 최대 3개
- 아코디언 최대 2개
- 스크롤은 하단 슬롯 1개만 허용
- 헤더/핵심요약/액션바는 항상 고정

## 8. 시연용 클릭 스토리보드

### 시연 순서
1. 메인 화면에서 현재 공장 상황 한 줄 요약
2. `운영 리스크 상황판` 클릭
3. `운영 리스크 상세 모달`에서 영향 라인과 리스크 구성 설명
4. `라인별 위험 우선순위 맵`에서 `LINE B` 클릭
5. `라인별 위험 상세 모달`의 `병목` 또는 `품질 영향` 탭 설명
6. `즉시 조치 상세 모달`로 이동
7. 자원 배분 결론 제시

### 발표용 한 줄 설명 예시
- 첫 화면: `현재 공장은 LINE B 중심의 생산차질 리스크가 가장 크고, QA 재검 지연이 LINE D까지 확산되는 상황입니다.`
- 리스크 상세: `리스크는 다운타임 증가와 QA 재검 지연이 결합된 구조입니다.`
- 라인 상세: `특히 LINE B는 다운타임과 병목 영향이 동시에 크기 때문에 최우선 관리 대상입니다.`
- 즉시 조치 상세: `따라서 오늘의 첫 조치는 LINE B 병목 해소와 AOI-02 재검 리소스 추가입니다.`

### 관리자 화면의 강점이 드러나는 흐름
`현황 파악 → 위험 집중 위치 확인 → 근거 확인 → 실행 판단`

이 순서가 관리자 화면의 본래 목적을 가장 선명하게 보여준다.

## 9. 개발/기획 전달용 최종 실행 규칙

### FE 기준 블록 분리 원칙
- 메인 화면 블록
  - 성과 KPI 3개
  - 운영 리스크 상황판
  - 즉시 조치 우선순위
  - 라인별 위험 우선순위 맵
  - 상태·알람 요약
- 상세 모달 블록
  - 성과 KPI 상세 모달
  - 운영 리스크 상세 모달
  - 즉시 조치 상세 모달
  - 라인별 위험 상세 모달
  - 알람 상세 모달
  - 상태 보드 상세 모달

### 공통 컴포넌트화 가능한 부분
- `KPI 카드`
- `상태칩`
  - `RUN / IDLE / DOWN / MAINT`
  - `CRIT / WARN / INFO`
- `Risk Score 바`
- `우선순위 리스트 아이템`
- `모달 헤더`
- `모달 액션바`
- `탭 헤더`
- `아코디언 블록`
- `스크롤 슬롯 리스트`

### 모달 구현 규칙
- 헤더/핵심요약/액션바 고정
- 스크롤은 하단 슬롯 1개만 허용
- 탭은 최대 3개
- 아코디언은 최대 2개
- `주방식 1개 + 보조방식 1개`까지만 허용
- 클릭한 메인 블록과 직접 연결된 정보만 표시

### 과밀 방지 규칙
1. 메인 화면 블록 수는 5개 수준으로 제한
2. 메인 화면에 5행 이상 목록 금지
3. 메인 화면에 대형 상세표 금지
4. 메인 화면에 다중 추세 차트 금지
5. 보조 블록이 `운영 리스크 상황판`과 `라인별 위험 우선순위 맵`보다 더 강하게 보이지 않게 함

### 해상도 기준 고려 포인트
- `1920x1080`
  - 전체 무스크롤 고정
  - 하단 `상태·알람 요약` 2열 유지
- `1366x768`
  - KPI 카드 높이 축소
  - 즉시 조치 우선순위 문장 1줄 압축
  - 하단 보조 블록 높이 최소화
  - 라인별 위험 맵은 4행 고정 유지

### 최종 정리
이 개편안은 관리자 화면을 `많이 보여주는 보고 화면`에서 `빠르게 판단하고 클릭으로 근거를 여는 의사결정 상황판`으로 재정의한다.  
메인 화면은 판단에 필요한 5개 블록으로 고정하고, 세부 설명은 모달 체계로 분리한다.  
이 구조를 그대로 적용하면 FE 구현 단위, 시연 흐름, 정보 위계가 모두 정리된다.
