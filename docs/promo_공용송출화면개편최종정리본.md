## 1. 개편 배경과 문제 정의

### 기존 공용 송출화면의 문제
기존 공용 송출화면은 운영 화면의 축소판처럼 보일 가능성이 컸다.  
상태 보드, 이슈, 추세, 보조 정보가 한 화면에 함께 들어오면 공용 송출화면이 `누구나 즉시 이해하는 화면`이 아니라 `설명이 필요한 화면`이 된다.

### 왜 텍스트/표 중심 구조를 바꿔야 하는지
- 운영 회의실, 시연, 방문자 브리핑 상황에서는 읽는 시간보다 보는 시간이 짧다.
- 표, 로그, 상세 설명은 5초 안에 이해하기 어렵다.
- 공용 송출화면은 숫자와 색, 상태칩, 이슈 카드처럼 시각 구조가 먼저 읽혀야 한다.
- 분석형 정보가 많아질수록 역할별 화면 차이가 흐려진다.

### 공용 송출화면이 해야 하는 역할
공용 송출화면은 `지금 공장은 정상인가, 이슈는 무엇인가?`를 한눈에 보여주는 `요약형 완결 상황판`이어야 한다.  
이 화면은 작업자/QA/관리자처럼 조작하거나 판단을 깊게 이어가는 화면이 아니라, `조작 없이 자동 송출되는 공장 현황판`이다.

## 2. 개편 목표와 설계 원칙

### 공용 송출화면의 핵심 질문
- `지금 공장은 정상인가, 이슈는 무엇인가?`

### 첫 화면 목표
- 5초 안에 현재 공장 상태를 이해할 수 있어야 한다.
- 한 줄로 전체 상태를 말할 수 있어야 한다.
- 핵심 이슈 3개를 즉시 설명할 수 있어야 한다.

### 정보 우선순위 원칙
1. `대형 KPI 3개`로 전체 생산 상태를 먼저 보여준다.
2. `라인 상태 요약 보드`로 공장 운영 상태를 색으로 보여준다.
3. `Top 이슈 3`으로 현재 설명 포인트를 고정한다.
4. `단일 추세 1개`로 흐름만 보조한다.
5. `롤링 메시지`는 최하단 보조 정보로만 둔다.

### 무스크롤 원칙
- 무조건 한 화면 완결형 구조로 설계한다.
- 블록 수는 4~5개 수준으로 제한한다.
- 차트는 1개만 허용한다.
- 표, 로그, 상세 설명은 메인 화면 금지한다.
- 클릭 기반 상세 확장 구조를 두지 않는다.

### 시연/회의/방문자 전달 원칙
- 설명보다 구조가 먼저 읽혀야 한다.
- 발표자는 상단 KPI → 중앙 상태 → 우측 이슈 → 하단 추세 순으로 설명하면 끝나야 한다.
- 방문자는 별도 조작 없이도 “정상/주의/정지” 상태를 바로 이해할 수 있어야 한다.

## 3. 첫 화면 정보 구조 확정안

### 첫 화면에 남길 블록
- `공장 전체 상태 헤드라인`
- `Today Output`
- `Achievement`
- `Yield`
- `라인 상태 요약 보드`
- `금일 Top 이슈 3`
- `시간대별 생산 흐름`
- `롤링 메시지`

### 첫 화면에서 제거할 블록
- 상세 테이블
- 전체 로그
- 원인 분석 차트
- 복수 추세 차트
- 역할별 조치 가이드
- 클릭 의존 상세 UI
- 경고/알람 전체 목록
- Lot/Inspection 단위 상세
- 설비별 상세 로그

### 블록 우선순위
1. `공장 전체 상태 헤드라인 + 대형 KPI`
2. `라인 상태 요약 보드`
3. `금일 Top 이슈 3`
4. `시간대별 생산 흐름`
5. `롤링 메시지`

### 시선 흐름
- 상단: 공장 전체 헤드라인 + KPI
- 중앙 좌측: 상태 요약 보드
- 중앙 우측: Top 이슈 3
- 하단: 생산 흐름
- 최하단: 롤링 메시지

### 블록 간 관계
- 헤드라인은 전체 공장을 한 줄로 요약한다.
- KPI는 지금 생산이 목표 대비 어떤 상태인지 보여준다.
- 상태 요약 보드는 라인별 운영 상태를 시각적으로 보여준다.
- Top 이슈는 발표자가 반드시 말해야 할 포인트를 고정한다.
- 생산 흐름은 현재 흐름이 유지 중인지 짧게 보조한다.
- 롤링 메시지는 추가 안내만 담당한다.

## 5. 왜 이 구조가 공용 송출화면에 맞는지 해설

### 방문자/회의 참석자 기준 장점
- 글을 많이 읽지 않아도 공장 상태를 바로 이해할 수 있다.
- 상태 색상과 큰 숫자가 먼저 보이기 때문에 설명 없이도 의미가 전달된다.
- 이슈가 3건으로 제한돼 있어 시선이 분산되지 않는다.

### 시연 기준 장점
- 발표자는 첫 줄 헤드라인으로 전체 상황을 요약할 수 있다.
- KPI, 상태 보드, Top 이슈, 생산 흐름 순으로 설명하면 자연스럽다.
- 운영 화면과 달리 “상세 설명이 필요 없는 요약판”이라는 차이가 분명해진다.

### 관리자 화면과의 차이
- 관리자 화면은 `왜 위험한가`와 `어디에 자원을 넣을까`를 판단하는 구조다.
- 공용 송출화면은 `지금 상태가 어떤가`와 `핵심 이슈는 무엇인가`까지만 보여준다.
- 따라서 리스크 근거, 권고 상세, 모달 흐름이 공용 송출화면에는 들어오지 않는다.

### 분석형 화면과 구별되는 이유
- 상세 분석이 없다.
- 클릭 확장이 없다.
- 복수 차트가 없다.
- 표와 로그가 없다.
- 단일 추세와 상태 요약만 남는다.

## 6. 개발/기획 전달용 최종 실행 규칙

### FE 기준으로 남길 블록
- `공장 전체 상태 헤드라인`
- `Today Output KPI`
- `Achievement KPI`
- `Yield KPI`
- `라인 상태 요약 보드`
- `Top 이슈 3`
- `시간대별 생산 흐름`
- `롤링 메시지 스트립`

### 제거할 블록
- 상세 테이블
- 전체 로그
- 원인 추적형 상세
- 복수 분석 차트
- 역할별 조치 버튼
- 상세 클릭 유도 UI
- 전체 알람 목록
- 세부 설비 로그
- Lot/Inspection 단위 정보

### 시각화로 치환할 블록
- 텍스트 상태 설명 → `상태칩`
- 긴 라인 상태 목록 → `라인 상태 요약 보드`
- 이슈 텍스트 나열 → `Top 이슈 3 카드형 구조`
- 생산 설명 문장 → `단일 생산 흐름 차트`
- 운영 메시지 → `롤링 메시지 스트립`

### 과밀 방지 규칙 5개
1. 메인 화면 블록 수를 5개 수준으로 제한한다.
2. 차트는 1개만 유지한다.
3. Top 이슈는 3건을 넘기지 않는다.
4. 표/로그/상세 문장을 메인 화면에 올리지 않는다.
5. 중앙 상태 보드와 상단 KPI보다 더 복잡한 보조 블록을 만들지 않는다.

### 해상도 기준 고려 포인트
- `1920x1080`
  - 현재 구조를 그대로 무스크롤 고정
  - KPI 3개, 상태 보드, Top 이슈, 하단 생산 흐름 모두 유지
- `1366x768`
  - KPI 카드 높이 축소
  - Top 이슈 문장 수를 1줄로 제한
  - 롤링 메시지를 더 얇게 축소
  - 생산 흐름 차트 높이를 낮추고 가독성 유지

### 송출용 화면 특성상 주의할 사항
- 기본 전제는 `조작 없는 자동 송출형`
- 가독성이 애니메이션보다 우선
- 자동 갱신은 허용하되, 과한 모션 전환 금지
- 화면 깜빡임 금지
- 상태색 대비를 강하게 유지
- 글자 크기는 운영 화면보다 크게 설정
- 멀리서도 읽히는 수준으로 대비와 간격 확보

## 7. 공용 송출화면에 절대 넣지 말아야 할 요소

- 상세 테이블

- 분석형 다중 차트
- 역할별 조치 가이드
- 클릭 의존 상세 구조
- 설비별 상세 로그
- 원인 추적형 데이터
- Lot/Inspection 상세
- 알람 전체 목록
- 긴 설명 문장
- 모달 중심 구조
- 운영자/QA/관리자 전용 세부 정보

## 한 줄 최종 결론

공용 송출화면은 `큰 숫자 + 상태 요약 + Top 이슈 3 + 단일 추세`만 남긴 비인터랙티브 한 화면 완결형 전광판 구조로 고정해야 한다.
