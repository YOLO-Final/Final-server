# 로그인 모듈 아키텍처 — 흐름 설명

---

## 전체 흐름 개요

사용자가 브라우저를 열면 시작됩니다.
흐름은 **브라우저 → Nginx → FastAPI 서버 → DB** 순서로 진행됩니다.

---

## 1단계 — 브라우저 영역

사용자 앞에는 두 가지 로그인 방식이 있습니다.

### 비밀번호 로그인
- 사원번호와 비밀번호를 입력하고 버튼을 누르면 서버로 전송합니다.

### Face ID 로그인
- 카메라가 켜지면서 1.8초마다 자동으로 인증을 시도합니다.
- 원하면 수동으로 버튼을 눌러 즉시 시도할 수도 있습니다.

### 두 방식의 공통 결과
두 방식 모두 인증에 성공하면 같은 결과로 수렴합니다.

> 인증 성공 → 인증 정보를 브라우저 저장소에 저장 → 대시보드 화면으로 이동

### 로그인 후 세션 관리 (dashboard.js)
대시보드 화면에 도착한 이후에는 세션 브리지가 동작합니다.

- 토큰이 만료되면 자동으로 갱신해서 요청을 재시도합니다.
- 갱신마저 실패하면 인증 정보를 지우고 다시 로그인 화면으로 돌려보냅니다.
- 로그아웃은 브라우저 저장소만 지우고 로그인 화면으로 이동합니다. 서버에는 별도 통보가 없습니다.

### 브라우저 저장소
로그인 성공 직후 아래 정보가 저장됩니다.

| 저장 항목 | 설명 |
|---|---|
| 사원번호 | 로그인한 사용자 식별자 |
| 접근 토큰 | API 요청 시 사용하는 인증 토큰 |
| 갱신 토큰 | 접근 토큰 만료 시 재발급에 사용 |

---

## 2단계 — Nginx

브라우저가 요청을 보내면 Nginx가 받습니다.

- **화면 파일 요청** → Nginx가 직접 돌려줍니다.
- **API 요청** → 뒤쪽 FastAPI 서버로 넘깁니다.

---

## 3단계 — FastAPI 서버 영역

서버 안에는 네 개의 구성 요소가 있습니다.

### 요청 수신 창구 (router.py)
브라우저에서 오는 모든 로그인·인증 요청을 받아서 처리 담당자에게 넘기는 입구입니다.

| 요청 종류 | 내용 |
|---|---|
| 비밀번호 로그인 요청 | 사원번호 · 비밀번호 수신 |
| Face ID 로그인 요청 | 얼굴 이미지 수신 |
| 토큰 갱신 요청 | 갱신 토큰 수신 |
| 사용자 정보 조회 | 사원번호로 이름 조회 |
| 로그아웃 | 서버 기능은 있으나 현재 프론트 미사용 |

### 인증 처리 (service.py)
실제 인증 로직이 모두 여기에 있습니다.

**비밀번호 검증**
- 비밀번호가 맞는지 확인합니다.
- 5회 실패 시 계정을 잠급니다.
- 구형 비밀번호 형식도 자동으로 호환 처리합니다.

**Face ID 매칭**
- 촬영된 얼굴과 등록된 얼굴을 비교합니다.
- 유사도가 기준값 0.30 이상이면 인증을 통과시킵니다.
- 1명을 지정해서 비교하거나, 전체 등록자를 대상으로 비교합니다.

**토큰 갱신 처리**
- 갱신 토큰이 유효한지 확인합니다.
- 유효하면 새 접근 토큰을 발급합니다.

### 토큰 관리 (jwt.py)
토큰을 만들고 해석하는 역할입니다.

**토큰 발급 · 해석**
- 로그인 성공 시 접근 토큰과 갱신 토큰을 생성합니다.
- 요청마다 토큰을 해석해 사용자를 확인합니다.

**보호 API 인증 확인**
- 토큰 유효성, 계정 상태, 토큰 버전 일치 여부를 확인하는 문지기 역할입니다.
- 불일치 시 접근을 차단합니다.

> ※ 토큰 생성과 인증 확인이 한 파일에 함께 존재합니다. 추후 역할별 분리 검토가 필요합니다.

### 얼굴 인식 엔진 (vision/service.py)
Face ID 로그인 시에만 호출됩니다.

**얼굴 특징값 추출**
- 얼굴 이미지를 받아 InsightFace 모델로 고유 특징값(벡터)을 추출합니다.

**등록 얼굴과 비교**
- 추출한 특징값과 DB에 저장된 특징값을 비교합니다.
- 기준값 0.30 이상이면 인증을 통과시킵니다.
- 판단 결과에 따라 아래 네 가지 실패 유형을 반환합니다.

| 실패 유형 | 설명 |
|---|---|
| 얼굴 불일치 | 등록된 얼굴과 다름 |
| 미등록 사원 | 얼굴 정보가 DB에 없음 |
| 이미지 처리 오류 | 이미지 해석 실패 |
| 엔진 준비 실패 | 인식 엔진 초기화 오류 |

## 4단계 — PostgreSQL DB

서버의 인증 처리와 토큰 관리가 데이터베이스에 접근합니다.

### user_table
사원 계정 정보 전체가 저장됩니다.

| 컬럼 | 설명 |
|---|---|
| employee_no | 사원번호 |
| password_hash | 비밀번호 (암호화 저장) |
| id_active | 계정 활성 여부 |
| id_locked | 계정 잠금 여부 |
| login_fail_count | 로그인 실패 횟수 |
| token_version | 토큰 버전 (강제 만료 제어) |
| last_login | 마지막 로그인 시간 |
| name | 사원 이름 |
| role | 권한 |

### face_embedding_table
사원별 얼굴 특징값이 저장됩니다. Face ID 인증 시 이 테이블의 값과 촬영 특징값을 비교합니다.

| 컬럼 | 설명 |
|---|---|
| employee_no | 사원번호 |
| embedding_json | 얼굴 특징값 (벡터) |
