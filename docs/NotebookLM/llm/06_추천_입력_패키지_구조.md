# 추천 입력 패키지 구조

NotebookLM에는 자료를 한 번에 던지기보다, 역할별로 분리된 묶음으로 올리는 편이 해석 안정성이 높다.

## 권장 폴더/묶음 구성

### 01_개요
- 넣을 것: `README_for_NotebookLM.md`, `LLM_Module_Scope_Summary.md`
- 역할: 이 자료 묶음의 목적과 해석 기준을 먼저 고정한다.
- 이유: NotebookLM이 첫 단계에서 범위와 목적을 잘못 잡지 않게 하기 위해서다.

### 02_백엔드_핵심코드
- 넣을 것: `main.py`, `api/v1/router.py`, `llm/router.py`, `llm_chat_api.py`, `knowledge_service.py`, `agent_service.py`, `sample/agent.py`, `openai_service.py`
- 역할: 실제 구현 범위와 책임 경계를 판단하는 source of truth 묶음.
- 이유: 개념도보다 먼저 읽어야 하는 자료다.

### 03_프론트연계근거
- 넣을 것: `dashboard.js`, `configuration.html`, `auto.html`, 필요 시 `manual.html`, `iot.html`, `model.html`
- 역할: 운영 UI와 실제로 연결된 기능 범위를 판별한다.
- 이유: `구현 존재`와 `운영 연계`를 분리하는 핵심 묶음이다.

### 04_API_및_흐름근거
- 넣을 것: `llm_media_api.py`, `llm_question.py`, `llm_result_api.py`, `LLM_Module_API_Summary.md`, `llm_architecture_sequence.puml`
- 역할: API별 입력/출력과 대표 흐름을 정리한다.
- 이유: NotebookLM이 엔드포인트를 기능 의미와 함께 읽게 만들기 위해서다.

### 05_저장검색메모리
- 넣을 것: `knowledge_service.py`, `sample/agent.py`, `LLM_Module_Limitations_and_Cautions.md`
- 역할: 지식 저장소, 검색, 세션 메모리, 웹 검색 보강, fallback 구조를 이해시키는 묶음.
- 이유: 이 영역이 가장 오해가 많기 때문에 별도 분리가 유리하다.

### 06_현재범위와한계
- 넣을 것: `LLM_Module_Current_vs_Conceptual_Architecture.md`, `LLM_Module_Limitations_and_Cautions.md`, 운영 연계표
- 역할: 현재 제공 범위, 운영 미연계 범위, 조건부 기능을 명확히 한다.
- 이유: PM용 문서에서 과장을 막는 핵심 입력 세트다.

### 07_아키텍처자료
- 넣을 것: `llm_architecture_report.md`, `llm_architecture.d2`, `발표용_기술반영_아키텍처.d2`, `발표용_사용자흐름_정돈.d2`
- 역할: 전체 서비스 내 위치와 개념 흐름을 설명한다.
- 이유: 코드만으로 부족한 구조 설명을 보강한다.

### 08_PM재사용요약
- 넣을 것: `LLM_Module_PM_Reusable_Sentences.md`, 짧은 소개 문장, 한 줄 요약, 주의 문장
- 역할: PM 보고서 본문에 재사용 가능한 안전한 문장을 제공한다.
- 이유: NotebookLM이 과장 대신 보수적 표현을 우선 채택하게 한다.

### 09_부록
- 넣을 것: `sample/main.py`, sample UI 정적 파일, sample knowledge 예시, sample faiss index, 추가 발표용 도식
- 역할: 필요 시 참고하는 보조 근거.
- 이유: 본문 핵심 근거와 섞으면 오해 가능성이 높으므로 끝에 둔다.

## 왜 이렇게 나누는가
- 코드, 운영 연계 근거, 개념 자료, 사람이 만든 요약 문서는 성격이 다르다.
- 이 구분이 없으면 NotebookLM이 `개념도 = 현재 구현`, `샘플 경로 = 상용 구조`, `API 존재 = 운영 기능`으로 오해하기 쉽다.
- 따라서 `핵심 근거 -> 운영 연계 -> 흐름/한계 -> 개념도 -> 부록` 순서로 읽게 만드는 것이 가장 안전하다.

## NotebookLM 권장 읽기 순서
1. `01_개요`
2. `06_현재범위와한계`
3. `02_백엔드_핵심코드`
4. `03_프론트연계근거`
5. `04_API_및_흐름근거`
6. `05_저장검색메모리`
7. `07_아키텍처자료`
8. `08_PM재사용요약`
9. `09_부록`

## 패키지 사용 원칙
- 핵심 사실은 항상 코드와 프론트 호출 근거에서 먼저 확인한다.
- 개념도는 설명 보조 자료로만 사용한다.
- 부록 자료는 본문 핵심 범위를 확장하거나 과장하는 데 사용하지 않는다.
