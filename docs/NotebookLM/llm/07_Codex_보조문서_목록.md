# Codex가 먼저 만들어야 할 보조 문서

코드 파일만 업로드하면 NotebookLM이 맥락 없이 기능을 확장 해석할 수 있다. 아래 보조 문서는 그 오해를 줄이고, PM용 보고서에 필요한 서술 방향을 먼저 고정하기 위한 문서들이다.

## README_for_NotebookLM.md
- 왜 필요한가: 자료 묶음의 목적, 읽는 순서, 해석 원칙을 가장 먼저 알려 줘야 한다.
- 들어갈 내용: 이 자료의 목적, LLM 모듈 범위, 운영 연계 판단 원칙, 코드와 개념도의 우선순위 규칙, 과장 금지 원칙.
- 적정 길이: 1페이지.
- NotebookLM 활용 방식: 모든 자료를 읽기 전 기준 문서로 사용.

## LLM_Module_Scope_Summary.md
- 왜 필요한가: 모듈의 역할, 범위, 현재 구현 수준을 짧게 고정해 줘야 한다.
- 들어갈 내용: 모듈 역할, 운영 UI 연계 기능, 핵심 처리 흐름, 저장/검색 방식, 외부 의존성, 현재 한계.
- 적정 길이: 1~2페이지.
- NotebookLM 활용 방식: 최종 보고서 초안의 기본 골격으로 사용.

## LLM_Module_Current_vs_Conceptual_Architecture.md
- 왜 필요한가: 발표용 개념도와 실제 구현 사이 차이를 분리하지 않으면 과장 위험이 크다.
- 들어갈 내용: 현재 구현 아키텍처, 개념 아키텍처, 차이점 목록, 보고서 서술 시 주의 문구.
- 적정 길이: 1페이지.
- NotebookLM 활용 방식: 구조 설명 시 개념과 구현을 분리하는 기준으로 사용.

## LLM_Module_API_Summary.md
- 왜 필요한가: 엔드포인트의 중요도와 운영 연계 여부를 표준화해서 보여 주기 위해 필요하다.
- 들어갈 내용: API 경로, 기능, 입력/출력, 운영 UI 직접 사용 여부, 비고.
- 적정 길이: 1~2페이지.
- NotebookLM 활용 방식: 기능 목록과 현재 범위 설명의 근거 표로 사용.

## LLM_Module_Limitations_and_Cautions.md
- 왜 필요한가: 조건부 기능과 미연계 범위를 명시하지 않으면 NotebookLM이 `가능 = 제공 중`으로 해석할 수 있다.
- 들어갈 내용: API 키 의존성, 메모리 비영속성, FAISS 조건부 사용, 스캔 PDF 한계, 웹 검색 과장 금지, 운영 미확인 API.
- 적정 길이: 1페이지.
- NotebookLM 활용 방식: 한계/리스크/주의사항 섹션의 기준 문서로 사용.

## LLM_Module_PM_Reusable_Sentences.md
- 왜 필요한가: PM이 바로 가져다 쓸 수 있는 문장을 먼저 제시하면 NotebookLM의 표현이 안정적이다.
- 들어갈 내용: 모듈 소개 문장, 기능 설명 문장, 운영 연계 문장, 한계 문장, 주의 문장.
- 적정 길이: 반 페이지~1페이지.
- NotebookLM 활용 방식: 최종 보고서 본문 문장 생성 시 보수적 표현 라이브러리로 사용.

## LLM_Module_Operational_Linkage_Table.md
- 왜 필요한가: 운영 연계 여부를 한눈에 볼 수 있는 별도 자료가 있으면 구현과 운영을 더 명확히 분리할 수 있다.
- 들어갈 내용: 기능/API, 백엔드 구현 여부, 프론트 호출 근거, 운영 상태, 비고.
- 적정 길이: 1페이지 표 중심.
- NotebookLM 활용 방식: `운영 중`, `미연계`, `구현만 존재` 라벨의 기준 표로 사용.

## LLM_Module_Source_Map.md
- 왜 필요한가: 어떤 파일이 source of truth인지 먼저 알려 주면 NotebookLM이 샘플이나 개념도를 과하게 쓰지 않는다.
- 들어갈 내용: 파일 분류, 우선순위, 사용 목적, 주의점.
- 적정 길이: 1페이지.
- NotebookLM 활용 방식: 원본 파일 해석 우선순위 지도 역할.
