# NotebookLM 오해 방지 가이드

이 문서는 NotebookLM이 LLM 모듈 자료를 과장 없이 해석하도록 만드는 보수적 가이드다.

## 반드시 넣어야 할 주의 문구
- 코드에 있는 기능이 모두 운영 중인 것은 아니다.
- 운영 연계 여부는 프론트엔드 호출 근거가 있는 경우에만 `운영 연계됨`으로 표시한다.
- 개념 아키텍처 그림과 현재 구현 아키텍처는 별도로 다뤄야 한다.
- `sample` 경로는 전부 데모 전용이 아니다. `sample/agent.py`는 실제 핵심 오케스트레이션 근거이며, `sample/main.py`와 `sample/static`은 데모 성격일 수 있다.
- 세션 메모리는 프로세스 메모리 기반일 수 있으며, DB 영속 저장으로 해석하면 안 된다.
- 웹 검색은 조건부 최신성 보강 기능이지, 실시간 최신성 보장 기능으로 쓰면 안 된다.
- PostgreSQL, Vector DB, 외부 검색 도구가 그림에 등장하더라도, LLM 모듈의 직접 사용 여부는 코드 기준으로 판단한다.

## 과장 방지 기준
- `구현됨`: 백엔드 코드나 실제 로직이 존재함.
- `운영 연계됨`: 프론트엔드 호출 또는 실제 서비스 흐름 근거가 확인됨.
- `조건부 사용 가능`: 환경 변수, API 키, 패키지 설치, 외부 네트워크 등 특정 조건이 충족될 때 사용 가능함.
- `개념도에만 존재`: 발표용/설명용 자료에 등장하지만 현재 코드로 직접 확인되지 않음.

## 구현 범위와 운영 범위를 구분하는 방식
- 백엔드 API가 존재해도 프론트엔드 호출 근거가 없으면 `구현 존재, 운영 미확인`으로 적는다.
- 프론트엔드에서 실제 호출되면 `운영 연계됨`으로 적는다.
- 운영 UI에 보이지 않아도 데모 경로에서만 쓰이는 경우 `데모/검증용 가능성`으로 적는다.

## 개념 아키텍처와 현재 구현 아키텍처를 구분하는 방식
- 코드 파일과 코드 추적 메모는 `현재 구현 아키텍처` 근거로 본다.
- D2/PUML/발표용 그림은 `개념 또는 설명용 아키텍처`로 본다.
- 둘이 다르면 보고서에는 `현재 구현 기준`과 `개념 설명 기준`을 분리해서 서술한다.

## sample 경로 해석 기준
- `server_api/src/modules/llm/sample/agent.py`: 실제 핵심 오케스트레이션 근거로 사용 가능.
- `server_api/src/modules/llm/sample/main.py`: 별도 데모 실행 경로로 본다.
- `server_api/src/modules/llm/sample/static/*`, `templates/*`: 샘플 UI 보조 자료로 본다.

## 세션 메모리 해석 기준
- 세션 상태가 agent 내부 메모리 구조에 유지되면 `프로세스 메모리 기반`으로 표기한다.
- 별도 DB 저장 코드가 확인되지 않으면 `영속 저장`으로 쓰지 않는다.

## 웹 검색 해석 기준
- 최신성 질의에서 웹 검색을 시도하거나 보강할 수 있어도 `조건부 최신성 보강`으로 적는다.
- 외부 검색 실패 가능성, API 키 필요성, 캐시 사용 여부가 있으면 반드시 함께 적는다.
- `실시간`, `보장`, `항상 최신` 같은 표현은 피한다.

## 저장소 및 외부 도구 해석 기준
- PostgreSQL: 전체 서비스 저장소일 수 있으나, LLM 모듈이 직접 읽고 쓰는지 코드로 별도 확인해야 한다.
- FAISS: 임베딩과 패키지 조건이 충족될 때 사용 가능한 검색 계층이다.
- Serper/Tavily/OpenAI Web Search: 선택적 또는 조건부 외부 검색 수단이다.
- Gemini/vLLM: 코드상 provider 옵션일 수 있으나 실제 운영 활성화 여부는 별도 확인이 필요하다.

## NotebookLM 작성 톤 가이드
- `지원한다`, `연계되어 있다`, `조건부로 사용할 수 있다`, `현재 코드 기준 확인된다` 같은 보수적 표현을 사용한다.
- `완전히 통합되어 있다`, `실시간 보장`, `영구 저장한다`, `운영 중이다` 같은 단정 표현은 근거가 있을 때만 사용한다.
