# 추가 요청

## NotebookLM 업로드용 파일명 규칙
- 숫자 접두어를 붙여 읽는 순서를 고정한다. 예: `01_`, `02_`, `03_`.
- 파일명에는 역할이 드러나게 쓴다. 예: `Scope`, `API_Summary`, `Limitations`, `Current_vs_Conceptual`.
- 개념 자료와 현재 구현 자료는 제목에 반드시 상태를 드러낸다.
- 예시 규칙:
  - `01_README_for_NotebookLM.md`
  - `02_LLM_Module_Scope_Summary.md`
  - `03_LLM_Module_API_Summary.md`
  - `04_LLM_Module_Current_vs_Conceptual_Architecture.md`
  - `05_LLM_Module_Limitations_and_Cautions.md`

## PM용 최종 보고서 작성을 위해 NotebookLM에 던질 핵심 질문 10개
1. 이 LLM 모듈은 전체 서비스 안에서 어떤 역할을 담당하는가?
2. 현재 운영 UI에서 실제로 연결되어 사용 중인 LLM 기능은 무엇인가?
3. 백엔드에 구현은 있으나 운영 연계가 확인되지 않은 기능은 무엇인가?
4. 사용자 질문이 들어왔을 때 답변이 생성되기까지의 실제 처리 흐름은 어떻게 되는가?
5. 문서 기반 질의응답에서 지식 문서는 어디에 저장되고 어떤 방식으로 읽히는가?
6. FAISS 검색은 항상 사용되는가, 아니면 조건부 fallback 구조인가?
7. 세션 메모리는 어떤 경계에서 유지되며 어떤 한계가 있는가?
8. 외부 의존성은 무엇이며 어떤 조건에서 기능이 축소되거나 실패할 수 있는가?
9. 개념 아키텍처와 현재 구현 아키텍처 사이 차이는 무엇인가?
10. PM이 전체 서비스 보고서에 바로 재사용할 수 있는 보수적인 모듈 소개 문장은 무엇인가?

## Codex가 NotebookLM에 함께 붙여 넣을 안내문 초안
```text
이 자료 묶음은 /server_api/src/modules/llm 기준으로 PM 제출용 LLM 모듈 보고서를 작성하기 위한 근거 자료입니다.

목적:
- PM의 전체 서비스 보고서에 재사용 가능한 LLM 모듈 설명 자료를 만들기 위한 입력 세트입니다.
- 포트폴리오용 자기 PR 문서가 아니라, 역할/연결 구조/현재 구현 범위/운영 연계/한계를 정확히 전달하는 것이 목적입니다.

해석 원칙:
1. 코드에 있는 기능이 모두 운영 중이라고 가정하지 마세요.
2. 운영 사용 여부는 프론트엔드 호출 근거와 함께 판단하세요.
3. 개념 아키텍처 그림과 현재 구현 코드는 분리해서 읽으세요.
4. sample 디렉토리의 모든 파일이 데모 전용은 아닙니다. sample/agent.py는 실제 핵심 오케스트레이션 근거입니다.
5. 세션 메모리는 프로세스 메모리 기반일 수 있으므로 DB 영속 저장으로 쓰면 안 됩니다.
6. 웹 검색은 조건부 최신성 보강 기능이지, 실시간 최신성 보장을 뜻하지 않습니다.
7. PostgreSQL, FAISS, 외부 검색도구는 그림에 등장하더라도 실제 직접 사용 여부를 코드 기준으로 판단하세요.

과장 금지 원칙:
- `구현됨`, `운영 연계됨`, `조건부 사용 가능`, `개념도에만 존재`를 구분해서 쓰세요.
- 코드에 없는 동작, 운영 연결, 저장 구조를 상상해서 채우지 마세요.

우선 읽어야 할 문서 순서:
1. README_for_NotebookLM.md
2. LLM_Module_Scope_Summary.md
3. LLM_Module_Current_vs_Conceptual_Architecture.md
4. LLM_Module_API_Summary.md
5. LLM_Module_Limitations_and_Cautions.md
6. 그 다음 원본 코드와 아키텍처 자료

작성 원칙:
- PM용 보고서 작성을 돕는 보수적 기술 설명을 우선합니다.
- 함수 단위 구현 해설이나 프롬프트 전문보다 역할, 연결 구조, 현재 범위와 한계를 중심으로 정리하세요.
```
