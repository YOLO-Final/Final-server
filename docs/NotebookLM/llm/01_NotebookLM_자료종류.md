# NotebookLM에 넣어야 할 자료 종류

이 문서는 `/home/orugu/Docker/Final_project_rss`의 LLM 모듈 관련 자료를 NotebookLM에 업로드할 때, 어떤 성격의 자료를 왜 넣어야 하는지 먼저 정리한 문서다. 목적은 PM 최종 보고서를 직접 쓰는 것이 아니라, NotebookLM이 과장 없이 PM용 모듈 설명 자료를 생성하도록 입력 맥락을 안정화하는 것이다.

## 1. 핵심 코드 근거 파일
- 왜 필요한지: 실제 구현 범위와 책임 경계를 확인하는 최우선 근거다.
- PM 문서 기여 지점: 모듈 역할, 처리 흐름, 저장 경계, 외부 의존성.
- 포함 예시: `server_api/src/main.py`, `server_api/src/api/v1/router.py`, `server_api/src/modules/llm/router.py`, `llm_chat_api.py`, `knowledge_service.py`, `sample/agent.py`.
- 역할: 개념도보다 우선하는 source of truth다.

## 2. 운영 UI 연계 근거 파일
- 왜 필요한지: 구현된 기능과 실제 운영 화면에서 사용되는 기능을 분리하기 위해 필요하다.
- PM 문서 기여 지점: 운영 연계 범위, 실사용 기능, 사용자 흐름.
- 포함 예시: `nginx/html/js/dashboard.js`, `nginx/html/configuration.html`, `nginx/html/auto.html`.
- 역할: `운영 중`과 `백엔드 구현 존재`를 구분하는 근거다.

## 3. API 및 입출력 설명 자료
- 왜 필요한지: NotebookLM이 엔드포인트 나열에 그치지 않고, 입력과 출력의 의미를 이해하게 해 준다.
- PM 문서 기여 지점: 기능 설명, API 요약, 사용자 요청-응답 구조.
- 포함 예시: `llm_chat_api.py`, `llm_media_api.py`, `llm_question.py`, `llm_result_api.py`.
- 역할: 채팅, STT/TTS, 추천 질문, 업로드/지식 관리의 경계를 설명한다.

## 4. 저장·검색·메모리 경계 자료
- 왜 필요한지: RAG, 파일 저장소, FAISS, 세션 메모리의 실제 동작 조건을 분리해야 과장을 막을 수 있다.
- PM 문서 기여 지점: 문서 저장/검색 방식, 세션 메모리 경계, 운영 제약.
- 포함 예시: `knowledge_service.py`, `sample/agent.py`.
- 역할: 파일 기반 지식, 조건부 FAISS, 프로세스 메모리 기반 세션을 설명한다.

## 5. 아키텍처 설명용 자료
- 왜 필요한지: 코드만으로는 전체 서비스 내 위치를 빠르게 파악하기 어렵다.
- PM 문서 기여 지점: 서비스 내 위치, 계층 구조, 개념 흐름.
- 포함 예시: `docs/architecture/llm_architecture_report.md`, `llm_architecture.d2`, `llm_architecture_sequence.puml`, `발표용_기술반영_아키텍처.d2`, `발표용_사용자흐름_정돈.d2`.
- 역할: 단, 개념 자료와 현재 구현 자료를 반드시 분리해서 읽게 해야 한다.

## 6. 한계·주의사항 자료
- 왜 필요한지: NotebookLM이 코드 존재를 곧 운영 보장으로 과장하는 것을 막기 위해 필요하다.
- PM 문서 기여 지점: 현재 범위, 미연계 기능, 조건부 기능, 리스크.
- 포함 예시: 사람이 먼저 작성한 limitation 문서, 운영 연계 표, 현재 구현 vs 개념 차이 정리.
- 역할: PM 문서의 톤을 보수적으로 고정한다.

## 7. PM 재사용용 요약 텍스트
- 왜 필요한지: 코드와 개념도만 넣으면 NotebookLM이 서술 중심을 잃기 쉽다.
- PM 문서 기여 지점: 모듈 소개 문장, 한 줄 요약, 기능 설명 문장.
- 포함 예시: `LLM_Module_Scope_Summary.md`, `LLM_Module_PM_Reusable_Sentences.md`.
- 역할: PM이 바로 가져다 쓸 수 있는 보수적 문장을 우선 학습시키는 자료다.

## 8. 보조 캡처 및 예시 자료
- 왜 필요한지: 화면과 흐름 예시는 NotebookLM이 사용자 관점 설명을 구성할 때 도움이 된다.
- PM 문서 기여 지점: 화면 설명, 사용자 흐름, 부록 예시.
- 포함 예시: 운영 화면 캡처, 대표 질문-응답 예시, 업로드/지식 상태 응답 예시.
- 역할: 본문 핵심은 아니지만 설명을 안정적으로 만든다.

## 자료별 역할 정리
- 코드 파일: 실제 구현 범위를 판단하는 1차 근거.
- UI 파일: 실제 운영 연결 여부를 판단하는 1차 근거.
- 개념도/아키텍처 자료: 구조를 이해시키는 2차 보조 근거.
- 요약 문서: NotebookLM의 서술 방향을 통제하는 가이드.
- 한계 문서: 과장과 오해를 막는 안전장치.

## 업로드 원칙
- 파일을 많이 넣는 것보다, 역할이 분리된 자료를 맥락과 함께 넣는 것이 우선이다.
- 코드와 개념도 중 충돌이 있으면 코드 기준으로 해석한다.
- 운영 연계는 프론트엔드 호출 근거가 있는 경우에만 `운영 연계`로 쓴다.
- `구현됨`, `운영 연계됨`, `조건부 사용 가능`, `개념도에만 존재`를 구분해서 설명한다.
