# 9.2.2 역할별 화면 목적 정리

## 핵심 요약

본 문서는 스마트 팩토리 불량 검출 시스템의 현장 대응, 데이터 분석, 통합 관제, 성과 공유를 기준으로 각 화면의 존재 목적과 사용자의 의사결정 포인트를 정의한 현황 정리서이다.

---

## 1. 작업 목적

- **사실 관계 정리**: 현재 구현된 시스템 상태를 기준으로 화면의 목적을 명확히 정의한다.
- **판단 기준 수립**: 단순 기능 나열이 아닌, 사용자가 화면을 통해 어떤 판단을 내리고 조치해야 하는지에 집중한다.
- **설계 기반 마련**: 정의된 목적을 바탕으로 후속 단계인 공용 레이아웃 및 탭 흐름 설계의 기준을 마련한다.

---

## 2. 대상 역할 및 화면 정의

문서 내 혼용되던 명칭을 화면 성격 중심으로 표준화하여 다음과 같이 구분한다.

| 화면 ID | 표준 화면 명칭 | 접근 대상 | 주요 업무 성격 | 비고 |
|---------|----------------|---------|--------------|------|
| SCR-001 | 현장 대시보드 | 작업자 | 실시간 상태 모니터링 및 즉각 대응 | 현장 대응 중심 |
| SCR-002 | 품질 대시보드 | 품질관리자 | 불량 분석 및 품질 개선 판단 | 데이터 분석 중심 |
| SCR-003 | 통합 대시보드 | 관리자 | 종합 지표 관리 및 운영 의사결정 | 통합 관제 중심 |
| SCR-004 | 생산 현황 보드 | 공용 송출 뷰 | 핵심 KPI 요약 및 성과 공유 | 성과 공유 중심 |

---

## 3. 역할별 주요 질문

각 대시보드는 사용자가 다음과 같은 핵심 질문에 답할 수 있도록 구성한다.

### 3.1 현장 대시보드 (작업자)

- 담당 라인의 작업지시가 계획대로 진행되고 있는가?
- 지금 바로 대응해야 할 불량 알람이 발생했는가?
- 현재 생산 속도로 오늘 목표량을 달성할 수 있는가?

### 3.2 품질 대시보드 (품질관리자)

- 불량이 집중적으로 발생하는 공정이나 라인은 어디인가?
- 가장 시급하게 개선이 필요한 주요 불량 유형은 무엇인가?
- 불량 판정 결과와 실제 이미지가 일치하는가?

### 3.3 통합 대시보드 (관리자)

- 전체 라인의 OEE와 목표 달성률이 안정적인가?
- 자재 부족 등 운영 차질이 발생할 리스크가 있는가?
- 업무 종료 시점의 예상 실적이 목표를 충족하는가?

---

## 4. 역할별 화면 목적 및 의사결정 포인트

단순 정보 조회를 넘어, 화면 확인 후 이어지는 사용자의 판단과 후속 행동을 정의한다.

| 화면 명칭 | 정제된 화면 목적 | 주요 판단 사항 | 후속 행동 |
|---------|-----------------|-------------|---------|
| 현장 대시보드 | 실시간 생산 상태와 알람을 확인하여 즉시 대응 여부를 판단한다. | 현재 발생한 알람 중 최우선 조치 대상은 무엇인가? | 알람 확인, 긴급 조치, 설비 점검 |
| 품질 대시보드 | 불량 유형 분포와 추이를 분석하여 개선 우선순위를 판단한다. | 불량이 집중되는 공정은 어디며 판정 결과가 타당한가? | 원인 분석, 재검 요청, 개선 대책 수립 |
| 통합 대시보드 | 종합 지표와 예측 정보를 파악하여 운영 자원 배분 방향을 판단한다. | 목표 달성을 저해하는 병목은 어디며 자원을 어디에 집중할 것인가? | 자원 재배치, 계획 조정, 업무 조율 |
| 생산 현황 보드 | 핵심 KPI와 당일 생산 성과를 공용 공간에서 한눈에 확인한다. | 현재 생산 및 품질 수준이 목표 대비 어느 정도인가? | 성과 공유, 현장 공감대 형성 |

---

## 5. 공용 화면과 역할 전용 화면 구분

권한과 사용 목적에 따라 화면의 데이터 노출 범위와 인터랙션 방식을 구분한다.

### 공용 화면 (생산 현황 보드)

- 원거리 시인성을 고려하여 대형 KPI 카드와 단순화된 차트 중심으로 구성한다.
- 입력 요소가 없는 조회 전용 특수 뷰 모드로 정의한다.

### 역할 전용 대시보드

- 상세 로그, 분석 데이터, 예측 정보 등 의사결정에 필요한 심화 정보를 포함한다.
- 알람 확인, 재검 요청 등 후속 조치를 위한 인터랙션 기능을 포함한다.

# 9.2.3 공용 레이아웃 및 컴포넌트 정리 (리뷰용)

## 핵심 요약
본 문서는 스마트 팩토리 불량 검출 시스템의 시각적 일관성과 개발 효율성을 높이기 위해, 현재 구현된 화면에서 확인되는 공통 구조, 공용 컴포넌트 후보, 상태 표기 규칙을 통합 정리한 리뷰용 문서이다.

---

## 1. 작업 목적

- 시각적 일관성 확보: 역할별로 분산된 화면의 레이아웃과 디자인 톤을 공통 기준으로 정리한다.
- 개발 효율성 증대: 반복 사용되는 UI 요소를 공용 컴포넌트 후보로 분류하여 재사용 범위를 검토한다.
- 의사결정 판단 근거 마련: 사용자가 화면에서 정보를 직관적으로 해석하고 후속 행동을 결정할 수 있도록 공통 규칙을 정리한다.

---

## 2. 공용 레이아웃 정의

현재 확인된 화면들은 정보의 중요도에 따라 요약 → 분석 → 상세로 이어지는 수직적 위계 구조를 공유하는 것으로 보인다.

| 영역 구분 | 주요 구성 요소 | 역할 및 특징 | 스크롤 여부 |
|---|---|---|---|
| 글로벌 헤더 (GNB) | 서비스 로고, 역할별 탭 메뉴 | 서비스 정체성 유지 및 역할별 화면 전환 | 상단 고정 |
| 핵심 지표 영역 (KPI) | 데이터 카드 | 생산량, 불량률 등 핵심 수치의 즉시 확인 | 본문 상단 |
| 시각화 영역 | 차트 및 그래프 | 시간대별 추이, 불량 분포 등 데이터 시각화 | 본문 중단 |
| 상세 정보 영역 | 리스트, 테이블, 로그 | 실시간 NG 내역, 검출 리스트 등 세부 데이터 확인 | 본문 하단 |

---

## 3. 공용 핵심 컴포넌트 목록

현재 확인된 화면에서 반복적으로 등장하는 요소를 기준으로 공용 컴포넌트 후보를 정리하면 다음과 같다.

### 3.1 전 화면 공통
- 글로벌 헤더(GNB)
- 역할 전환 탭 메뉴
- LIVE 인디케이터

### 3.2 역할 기반 화면 공통
- KPI 데이터 카드
- 차트 카드
- 실시간 로그 리스트 또는 상세 데이터 테이블
- 상태 배지

### 3.3 역할별 특화 요소
- 현장 대시보드: 긴급 알람 배너, 최근 10분 NG 로그
- 품질 대시보드: 불량 유형 분포 차트, 재검 판단 테이블
- 통합 대시보드: 예측 불량률/종료량 KPI, 라인별 성과 비교 패널
- 생산 현황 보드: 전시형 대형 KPI 카드

---

## 4. 상태 표현 및 데이터 표기 규칙

정보의 오인지를 줄이기 위해 색상과 수치 표기 방식의 공통 기준을 다음과 같이 정리하였다.

### 4.1 상태별 색상 체계
- Red: 위험, 정지, 긴급 알람 등 즉각 조치가 필요한 상태
- Yellow: 경고, 주의, 재검 필요 등 관찰 및 확인이 필요한 상태
- Green: 정상, 완료, 안정 상태
- Blue: 진행 중, 분석 중, 조치 수행 중 상태
- Gray: 대기, 점검, 확인 전, 로딩 대기 상태

### 4.2 데이터 표기 표준
- 날짜/시간: 전체 일시는 `YYYY-MM-DD HH:mm:ss` 형식을 기본으로 사용
- 비율: 소수점 첫째 자리까지 표기  
  예: `98.5%`
- 수량: 천 단위 구분 기호 적용  
  예: `1,250 EA`

---

## 5. 역할별 적용 및 특화 요소

공통 레이아웃을 기반으로 하되, 역할별 업무 성격에 따라 특화 컴포넌트를 다르게 배치하는 방향으로 정리하였다.

| 화면 명칭 | 업무 성격 | 특화 컴포넌트 |
|---|---|---|
| 현장 대시보드 | 즉시 대응 | 긴급 알람 배너, 최근 10분 NG 로그 |
| 품질 대시보드 | 데이터 분석 | 불량 유형 분포 차트, 재검 판단 테이블 |
| 통합 대시보드 | 운영 의사결정 | 예측 불량률/종료량 KPI, 라인별 성과 비교 패널 |
| 생산 현황 보드 | 성과 공유 | 전시형 대형 KPI 카드 |

생산 현황 보드는 공통 골격을 일부 공유하되, 일반 업무형 화면과 구분되는 전시형 특수 화면으로 보는 것이 적절하다.

---

## 6. 후속 설계 반영 및 추가 확인 필요 항목

현재 단계에서 미확정된 항목은 후속 설계 과정에서 추가 정리가 필요하다.

- 조회 조건(Filter) 구조: 기간, 라인, 모델 선택 필터 영역의 배치 위치와 동작 방식
- 상세 이동(Drill-down) 방식: 카드/차트 클릭 시 상세 페이지 이동 또는 모달/팝업 호출 여부
- 권한 및 로그인: 역할별 메뉴 노출 범위와 초기 진입 화면 정책
- 용어 및 명칭 통일: ‘홍보’, ‘전시’, ‘생산 현황 보드’ 등 명칭 정리
- 상태값 세부 적용 범위: 설비 상태, 업무 상태, 위험도 상태 간 구분 기준

---

## 7. 정리

현재 화면들은 상단 GNB와 KPI 중심의 공통 레이아웃 구조를 공유하며, 차트와 상세 정보 영역이 역할별 목적에 따라 변형 적용되는 것으로 정리되었다. 본 문서는 후속 단계인 역할별 탭 흐름 정리와 공용 컴포넌트 우선 개발 범위 설정의 기준 문서로 활용할 수 있다.

# 9.5.1-7 화면별 KPI 최종 구성안

## 핵심 요약
본 문서는 스마트 팩토리 불량 검출 시스템의 각 대시보드 목적에 맞추어 KPI를 화면별로 3~6개 수준으로 압축하고, 역할별 의사결정에 최적화된 최종 KPI 세트를 정의한 결과입니다.  
본 최종 구성안은 **KPI 우선순위, 데이터 가용성, 화면 목적, 시인성, 즉시 대응 필요성**을 함께 고려하여 각 화면에 우선 배치할 KPI 세트를 확정한 기준안입니다.

---

## 1. 개요
화면별 KPI 최종 구성은 단순히 중요한 지표를 많이 배치하는 작업이 아니라,  
**각 역할이 해당 화면에서 가장 먼저 확인해야 하는 질문에 답할 수 있는 지표만 선별하여 상단에 배치하는 작업**입니다.

따라서 본 문서에서는 다음 기준을 적용합니다.

- 각 화면의 존재 목적에 직접 연결되는 KPI를 우선 반영한다.
- Must KPI를 중심으로 구성하되, 필요 시 Should KPI를 보조적으로 포함한다.
- 한 화면의 상단 KPI 카드는 과밀하지 않도록 4개 내외를 원칙으로 한다.
- 동일한 의미의 지표는 화면별 집계 범위만 달리 적용하고 계산 기준은 최대한 통일한다.
- 상세 원인 분석에 적합한 항목은 상단 카드보다 차트 또는 하단 리스트로 배치한다.

---

## 2. 화면별 KPI 최종 구성

### 2.1 현장 대시보드 (SCR-001)

- **목적**: 실시간 상태 모니터링 및 불량 급증 시 즉각 대응
- **대상**: 현장 작업자

현장 대시보드는 즉시 대응이 가장 중요하므로, 작업자가 화면 진입 직후  
**설비 상태 이상 여부**, **불량 급증 여부**, **현재 생산 진척도**를 한눈에 판단할 수 있도록 구성합니다.

| KPI명 | 선정 사유 | 관련 의사결정 | 중요도 |
|---|---|---|---|
| 최근 10분 NG 발생 건수 | 실시간 불량 급증 여부를 가장 빠르게 파악할 수 있음 | 라인 정지 및 긴급 점검 결정 | Must |
| 설비 가동 상태 | 현재 라인의 정상 동작 여부(RUN / DOWN / IDLE)를 즉시 확인 가능 | 즉각적인 설비 복구 조치 | Must |
| 목표 달성률 | 당일 계획 대비 현재 진척도를 직관적으로 확인할 수 있음 | 생산 속도 조절 및 잔업 판단 | Must |
| 누적 생산량 | 현재 작업 실적을 수량 기준으로 빠르게 확인할 수 있음 | 작업 진행 상황 및 생산 집중도 판단 | Must |

#### 구성 원칙
- 현장 화면은 **즉시 대응형 KPI**를 우선 배치합니다.
- `현재 불량률`은 중요하지만 작업자 화면에서는 상단 카드보다 **보조 품질 블록 또는 차트 영역**에 두는 것이 더 적절합니다.

---

### 2.2 품질 대시보드 (SCR-002)

- **목적**: 불량 원인 분석 및 재검/최종 판정 의사결정
- **대상**: 품질관리자(QA)

품질 대시보드는 현재 품질 수준, 주요 불량 원인, 사람이 추가 확인해야 할 대상을 빠르게 파악할 수 있도록 구성합니다.

| KPI명 | 선정 사유 | 관련 의사결정 | 중요도 |
|---|---|---|---|
| 현재 불량률 | 전체 검사 대비 누적 불량 비율을 빠르게 파악할 수 있음 | 품질 목표 달성 여부 판단 | Must |
| 주요 불량 유형 (Top 5) | 가장 빈번한 불량 원인을 식별할 수 있음 | 공정 개선 및 원인 분석 우선순위 결정 | Must |
| 재검 필요 수량 | AI 확신도가 낮아 수동 검토가 필요한 대상을 확인할 수 있음 | 작업 우선순위 및 인력 배분 | Must |
| 검사 현황 요약 (총 검사 수량 / 양품 수 / 불량 수) | 당일 전체 품질 상태를 총량 기준으로 파악할 수 있음 | 전체 품질 실적 보고 및 이상 여부 확인 | Must |
| 예측 불량률 (ML) | 자재/환경 변수 기반 향후 불량 추세를 보조적으로 참고할 수 있음 | 선제적 품질 사고 예방 | Should |

#### 구성 원칙
- 품질 화면은 **현재 상태 + 원인 분석 + 재검 판단**이 동시에 가능해야 합니다.
- `검사 현황 요약`은 단일 KPI가 아니라 **묶음형 요약 카드**로 해석합니다.
- `예측 불량률`은 초기 구현에서는 Should로 두고, 데이터 확보 이후 확대 적용합니다.

---

### 2.3 통합 대시보드 (SCR-003)

- **목적**: 전사 생산 지표 통합 관제 및 운영 자원 배분
- **대상**: 공장 관리자

통합 대시보드는 공장 전체의 효율, 계획 달성 가능성, 라인 간 편차를 함께 판단할 수 있도록 구성합니다.

| KPI명 | 선정 사유 | 관련 의사결정 | 중요도 |
|---|---|---|---|
| 전체 라인 OEE | 가동률, 성능, 품질을 통합한 종합 효율 지표 | 공장 전체 생산성 평가 | Must |
| 공장 전체 목표 달성률 | 전사 계획 대비 현재 실적 진척도를 한눈에 파악 가능 | 전사 생산 계획 조정 | Must |
| 예상 종료 생산량 | 현재 속도 유지 시 종료 시점의 실적을 예측할 수 있음 | 납기 준수 여부 및 자원 재배치 | Must |
| 라인별 생산량 비교 | 라인 간 성과 편차와 병목 구간을 식별할 수 있음 | 병목 해소 및 인력 지원 결정 | Must |
| 불량 위험도 예측 | 향후 단기 품질 리스크를 보조적으로 확인할 수 있음 | 선제적 리스크 관리 | Should |

#### 구성 원칙
- 관리자 화면은 **비교형 KPI**와 **예측형 KPI**를 함께 포함합니다.
- `공장 전체 목표 달성률`은 작업자 화면의 목표 달성률과 구분하기 위해 전사 집계 기준임을 명확히 합니다.

---

### 2.4 생산 현황 보드 (SCR-004)

- **목적**: 핵심 성과 공유 및 현장 공감대 형성
- **대상**: 전 사원 (공용 송출)

생산 현황 보드는 복잡한 분석보다, 공장 전체의 현재 상태를 멀리서도 한눈에 읽을 수 있도록 구성합니다.

| KPI명 | 선정 사유 | 관련 의사결정 | 중요도 |
|---|---|---|---|
| 누적 총 생산량 | 당일 공장 전체 성과를 대형 수치로 직관적으로 전달 | 목표 달성 의지 고취 | Must |
| 목표 달성률 | 계획 대비 현재 위치를 직관적으로 공유 | 생산 진척 상황 인지 | Must |
| 당일 양품률 | 공장 전체 품질 수준을 한눈에 전달 | 품질 우선 문화 형성 | Must |
| TOP 3 Defect Class | 현재 가장 주의해야 할 불량 유형을 요약하여 전파 | 품질 주의사항 현장 공유 | Must |

#### 구성 원칙
- 공용 보드는 **시인성**과 **즉시 이해 가능성**을 최우선으로 합니다.
- 세부 로그나 긴 설명보다는 숫자와 핵심 불량 유형 요약 중심으로 구성합니다.

---

## 3. 공용 KPI와 역할 특화 KPI 구분

시스템의 일관성과 재사용성을 높이기 위해 KPI는 공용 KPI와 역할 특화 KPI로 구분하여 관리합니다.

### 3.1 공용 KPI (Standard KPIs)
공용 KPI는 여러 화면에서 공통으로 사용하되, 화면별 집계 범위만 다르게 적용하는 표준 지표입니다.

- **목표 달성률**  
  - 계산 기준: `해당 화면의 집계 범위(라인/공장)에 대한 현재 양품 수량 ÷ 목표 수량 × 100`
- **현재 불량률**  
  - 계산 기준: `해당 화면의 집계 범위(라인/공장)에 대한 누적 불량 수량 ÷ 누적 검사 수량 × 100`
- **누적 생산량 계열 지표**
  - 작업자 화면: 라인 또는 작업 기준 누적 생산량
  - 관리자/공용 화면: 공장 전체 누적 생산량

### 3.2 역할 특화 KPI (Specialized KPIs)
역할 특화 KPI는 특정 역할의 핵심 의사결정에 직접 연결되는 전용 지표입니다.

- **현장 작업자**
  - 최근 10분 NG 발생 건수
  - 설비 가동 상태

- **품질관리자**
  - 재검 필요 수량
  - 주요 불량 유형 (Top 5)
  - 예측 불량률

- **공장 관리자**
  - 전체 라인 OEE
  - 예상 종료 생산량
  - 라인별 생산량 비교
  - 불량 위험도 예측

- **공용 보드**
  - TOP 3 Defect Class
  - 당일 양품률

---

## 4. 화면별 배치 원칙

### 4.1 상단 KPI 카드
- 작업자 / 품질 / 관리자 대시보드는 **상단 KPI 카드 4개 내외**를 기본으로 합니다.
- 공용 보드는 **대형 KPI 3개 + 보조 정보 1개** 구성을 원칙으로 합니다.
- 상단 KPI는 가능한 한 **즉시 상태 판단이 가능한 지표**를 우선 배치합니다.

### 4.2 중단 차트 영역
- 상단 KPI의 원인 분석이 필요한 항목은 중단 차트로 배치합니다.
- 추세형 KPI는 라인차트, 비교형 KPI는 막대차트, 구성형 KPI는 도넛/막대차트 중심으로 구성합니다.

### 4.3 하단 상세 리스트
- 상세 로그, 불량 상세 목록, 검사 이력, 알람 목록 등은 하단 리스트 또는 테이블로 배치합니다.
- 공용 전시 보드는 가독성을 위해 하단 상세 리스트를 최소화합니다.

---

## 5. 시각화 및 배치 가이드라인

한 화면에 정보가 과밀되지 않도록 다음 원칙을 적용합니다.

- **KPI 카드 (상단)**  
  핵심 수치를 가로 배열하며, 임계값에 따라 배경색 또는 텍스트 색상을 변경하여 상태 인지를 돕습니다.

- **차트 영역 (중단)**  
  KPI의 추세, 비교, 분포를 시각화하여 원인 분석과 상황 이해를 지원합니다.

- **상세 리스트 (하단)**  
  실시간 로그, 검사 대상, 상세 데이터 테이블을 배치하여 후속 확인이 가능하도록 합니다.

- **중복 최소화 원칙**  
  동일한 의미의 KPI를 한 화면 내에서 반복 배치하지 않고, 상단-중단-하단 역할을 구분합니다.

---

## 6. 정리
본 문서는 화면별 KPI 최종 묶음을 확정한 기준 문서로서, 이후 차트 유형 선정, UI 배치, API 응답 구성, 더미데이터 검증의 기준으로 활용됩니다.

따라서 후속 구현 단계에서는 단순히 많은 정보를 넣는 방향이 아니라,  
**각 화면의 목적에 맞는 핵심 KPI를 가장 먼저 보여주고, 나머지 정보는 분석 영역으로 분리하는 방향**을 유지해야 합니다.

즉, 화면별 KPI 최종 구성의 목적은  
**정보 나열이 아니라 역할별 판단 최적화**에 있습니다.

# 9.5.2 최종 차트 유형 선정서_(최종)

## 핵심 요약
본 문서는 스마트 팩토리 불량 검출 시스템의 4개 화면에 대해 KPI별 최종 시각화 유형을 확정하고, 그 선정 근거와 구현 우선순위, 데이터 요구사항 요약을 정리한 최종 기준서이다.  
본 문서는 앞서 수행된 KPI 성격 분류, 차트 후보안 도출, 화면별 배치 초안, 1차/2차 구현 범위 구분, 데이터 요구사항 정의 결과를 통합한 문서로서, 후속 단계인 **9.5.3 차트/테이블 구현**의 직접 기준으로 활용한다.

---

## 1. 작업 목적
최종 차트 유형 선정의 목적은 화면별 KPI에 대해 가장 적합한 시각화 유형을 확정하여, 사용자가 각 화면에서 필요한 판단을 가장 빠르고 정확하게 수행할 수 있도록 구현 기준을 정립하는 데 있다.

세부 목적은 다음과 같다.

- KPI별 최종 차트 유형을 확정한다.
- 1차 구현 대상과 2차 고도화 대상을 구분한다.
- 화면 목적, KPI 성격, 구현 가능성, 데이터 정합성을 종합 반영한다.
- 후속 구현 단계에서 화면 간 일관된 시각화 기준으로 활용한다.

---

## 2. 최종 선정 기준

### 2.1 기본 선정 원칙
최종 차트 유형은 아래 원칙을 기준으로 선정한다.

- **해석 최적화**: KPI의 성격(현황, 상태, 추세, 비교, 분포, 예측, 상세)에 가장 적합한 차트를 우선 적용한다.
- **역할별 위계**: 작업자(즉시 대응), 품질관리자(원인 분석), 관리자(성과 비교 및 운영 판단), 공용 보드(원거리 시인성)에 맞는 표현 구조를 적용한다.
- **구현 현실성**: 1차 구현은 현재 Chart.js 및 HTML/CSS 구조에서 안정적으로 구현 가능한 표준 차트를 우선 적용한다.
- **데이터 정합성**: 차트가 요구하는 최소 데이터 필드와 API 응답 구조를 일치시키는 방향으로 선정한다.

### 2.2 1차 / 2차 구분 원칙
- **1차 구현(Phase 1)**: KPI 카드, 상태 배지, 프로그레스 바 카드, 라인차트, 수평/수직 바차트, 도넛차트, 로그 리스트, 상세 테이블
- **2차 고도화(Phase 2)**: 게이지 차트, 자동 롤링 고도화, 점선형 예측 구간, 카드 애니메이션, 스파크라인 카드, 이미지 썸네일 확장, 복합 차트 고도화

### 2.3 적용 범위 유의사항
- 본 문서의 최종 차트는 **기본 시각화 기준**을 의미한다.
- 경고 배너, 자동 롤링, 점멸 효과, 카운트 업 애니메이션 등은 **보조 UI 요소 또는 고도화 연출 요소**로 관리한다.
- 구현 코드 위치 및 함수명은 참고 자료로 활용하되, 본 문서의 핵심은 **최종 시각화 의사결정 기준**에 둔다.

## 3. 화면별 최종 차트 선정 결과

## 3.1 현장 대시보드 (SCR-001)
현장 대시보드는 실시간 상태 모니터링과 즉시 대응이 핵심이므로, 상단에는 카드형/상태형 요소를, 하단에는 최근 이벤트와 보조 분석 정보를 배치한다.

| 화면 | KPI / 요소 | 최종 차트 | 대안·고도화 | 선정 근거 | 1차·2차 | 데이터 요구사항 요약 |
|---|---|---|---|---|---|---|

## 5. 구현 연계 참고 사항

본 문서의 최종 차트 유형은 현재 프론트엔드 구조를 기준으로 다음 원칙과 연계한다.

- 차트 엔진은 dashboard_charts.js의 Chart.js 기반 렌더링 구조를 표준으로 활용한다.
- 화면 조립은 dashboard_app.js의 역할별 렌더링 구조와 연결한다.
- 차트 데이터는 dashboard_api.js 기준 응답 구조와 일치하도록 맞춘다.
- 차트 재생성 시에는 기존 인스턴스를 정리하는 방식으로 메모리 누수를 방지한다.
- 자동 롤링, 카운트 업, 게이지 등 고도화 요소는 1차 기본 기능 안정화 이후 적용한다.

- 9.5.4 역할별 화면 배치 확정: 차트 유형과 화면 영역 배치 최종 연결

## 7. 결론

최종 차트 유형 선정은 단순히 보기 좋은 시각화를 정하는 작업이 아니라, 각 KPI를 해당 화면의 사용자 목적과 구현 현실성에 맞게 가장 효과적으로 전달할 수 있는 표현 방식을 확정하는 과정이다.
따라서 본 문서는 화면별 KPI 전체를 대상으로 최종 차트, 대안·고도화 방향, 선정 근거, 구현 단계, 데이터 요구사항 요약을 통합 정리하였으며, 후속 구현 단계에서 일관된 설계 기준으로 활용할 수 있도록 구성하였다.

# 9.5.3.1 차트/테이블 구현 대상 목록

## 핵심 요약

본 문서는 스마트 팩토리 불량 검출 시스템의 4개 화면을 대상으로, 차트/테이블 구현 대상을 화면별로 확정한 결과서이다. 구현 우선순위는 DB 1차 구축 범위 내 즉시 구현 가능 여부를 기준으로 구분하였으며, `production_logs`, `defect_logs`, `lines`, `product_models` 기반 항목은 1차 구현 대상으로, `equipment`, `notifications`, `ml_predictions` 등 2차 연계 테이블에 직접 의존하는 항목은 확인 필요 또는 2차 구현 대상으로 분류하였다.

## 1. 개요

본 문서는 9.5.1 KPI 정의서와 9.5.2 최종 차트 유형 선정서를 바탕으로, 실제 개발 단계에서 구현해야 할 시각화 요소를 화면별로 정리한 구현 기준 문서이다. 각 항목은 다음 기준으로 우선순위를 판정하였다:

- 화면 목적
- 사용자 의사결정 시급성
- 데이터 확보 가능성
- DB 설계 범위와의 정합성

특히 1차 구현은 `production_logs`, `defect_logs`, `lines`, `product_models` 중심으로 구성하고, 설비 상태, 알림, 예측 결과처럼 2차 테이블 연계가 필요한 항목은 별도 구분하여 후속 구현 계획과 충돌하지 않도록 정리하였다.

---

## 2. 구현 우선순위 분류 기준

### 2.1 1차 유지

다음 조건을 만족하는 항목은 1차 구현 대상으로 유지한다.

- `production_logs`, `defect_logs`, `lines`, `product_models` 기반으로 즉시 산출 가능한 항목
- 시간, 수량, 비율, 유형 분류 등 표준 집계 구조로 Chart.js 기반 렌더링이 가능한 항목
- 현장 대응, 품질 판정, 관리자 운영 판단에 직접 사용되는 핵심 KPI 항목

### 2.2 확인 필요

다음 조건에 해당하는 항목은 구현 대상에는 포함하되, 세부 산식 또는 데이터 소스를 추가 확인한 뒤 구현 여부를 최종 확정한다.

- `lines` 테이블만으로 일부 표현은 가능하나, 실제 운영 의미가 `equipment` 수준 정의와 충돌할 수 있는 항목
- DB 필드만으로는 계산 가능성이 있으나 산식 정의 또는 업무 기준 추가 합의가 필요한 항목
- 1차 구현으로 시도할 수 있으나, 범위 과다 또는 기준 미확정 위험이 있는 항목

### 2.3 2차 구현

다음 조건을 만족하는 항목은 2차 구현 대상으로 분류한다.

- `equipment`, `notifications`, `ml_predictions`, `environment_logs` 등 2차 테이블에 직접 의존하는 항목
- 예측 모델 안정화, 추가 라이브러리, 인터랙션 연출, 복합 계산이 필요한 항목
- 1차 핵심 운영지표 구현 이후 고도화 단계에서 적용하는 것이 적절한 항목

---

## 3. 화면별 차트/테이블 구현 대상 목록

### 3.1 현장 대시보드 (SCR-001)

현장 작업자의 즉각적인 대응을 지원하기 위해, 본 화면은 실시간 불량 현황과 생산 진행 상태를 우선 구현 대상으로 설정한다. 불량 발생 추이, 최근 로그, 생산 실적 등은 `defect_logs`, `production_logs`, `lines` 기반으로 구현 가능하므로 1차 유지 대상으로 분류한다. 반면 설비 상세 상태와 예측성 위험도는 `equipment` 또는 `ml_predictions` 연계가 필요하므로 별도 구분한다.

| 구분 | KPI/요소명 | 시각화 유형 | 구현 분류 | 비고 |
|------|-----------|-----------|---------|------|
| 카드 | 최근 10분 NG 발생 건수 | KPI 카드 (Red) | **1차 유지** | `defect_logs.recorded_at`, `defect_qty` 기반 집계 |
| 카드 | 목표 달성률 | 프로그레스 바 카드 | **1차 유지** | `production_logs.actual_qty`, `target_qty` 기반 |
| 카드 | 누적 생산량 | 대형 수치 카드 | **1차 유지** | 당일 누적 생산 실적 표시 |
| 리스트 | 실시간 NG 로그 | 로그 리스트 | **1차 유지** | 최신 불량 이벤트 중심 표시 |
| 차트 | 최근 발생 유형 Top 3 | 수평 바차트 | **1차 유지** | `defect_class` 기준 실시간 분포 |
| 카드 | 설비 가동 상태 | 상태 배지 (Badge) | **확인 필요** | `lines.status`로 단순 구현 가능하나, 실제 설비 상태는 `equipment.status`와 구분 필요 |
| 배너 | 긴급 알람 배너 | 텍스트 배너 | **2차 구현** | `notifications` 연계가 필요 |
| 예측 | 설비 이상 예측 | 위험도 점수 카드 | **2차 구현** | `ml_predictions` 또는 `equipment` 연계 필요 |

### 3.2 품질 대시보드 (SCR-002)

품질관리자의 원인 분석과 재검 판정을 지원하기 위해, 불량률 추세, 유형별 비교, 상세 이력 조회를 중심으로 구성한다. `defect_logs`는 불량 유형별 세부 조회와 시간대 집계에 적합하게 설계되어 있으므로 본 화면의 핵심 차트와 테이블 다수는 1차 구현이 가능하다. 다만 재검 여부나 AI 신뢰도처럼 별도 판정 기준이 필요한 항목은 추가 확인 또는 2차 구현으로 분류한다.

| 구분 | KPI/요소명 | 시각화 유형 | 구현 분류 | 비고 |
|------|-----------|-----------|---------|------|
| 카드 | 현재 불량률 | KPI 카드 | **1차 유지** | `inspection_qty`, `defect_qty` 기반 계산 |
| 카드 | 검사 현황 요약 | 묶음형 정보 카드 | **1차 유지** | 검사/양품/불량 수량 집계 가능 |
| 차트 | 주요 불량 유형 Top 5 | 수평 바차트 | **1차 유지** | 유형별 비교 분석 핵심 |
| 차트 | 시간대별 불량률 추이 | 라인차트 | **1차 유지** | `recorded_at` 시간 집계 가능 |
| 테이블 | NG 검출 리스트 | 상세 테이블 | **1차 유지** | 라인/유형/수량 단위 세부 조회 |
| 카드 | 재검 필요 수량 | KPI 카드 (Yellow) | **확인 필요** | 별도 재검 상태 필드 부재, 판정 기준 합의 필요 |
| 예측 | 예측 불량률 | 예측값 전용 카드 | **2차 구현** | `ml_predictions.pred_type = defect` 연계 필요 |
| 차트 | AI 판정 신뢰도 분포 | 도넛 차트 | **2차 구현** | 현재 DB 정의상 `confidence` 분포 필드 확인 필요 |

### 3.3 통합 대시보드 (SCR-003)

관리자의 운영 의사결정을 위해, 전체 라인 비교와 생산 추세 파악을 1차 목표로 한다. `production_logs`와 `defect_logs`는 라인별 합계, 시간대별 집계, 유형별 비교에 적합하므로 비교형 차트와 추세 차트는 1차 구현 대상으로 유지한다. 반면 OEE, 예측 생산량, 위험도 요약과 같은 항목은 추가 산식 또는 예측 테이블 연계가 필요하므로 별도 구분한다.

| 구분 | KPI/요소명 | 시각화 유형 | 구현 분류 | 비고 |
|------|-----------|-----------|---------|------|
| 카드 | 공장 목표 달성률 | 프로그레스 바 카드 | **1차 유지** | 전사 기준 진척률 계산 |
| 차트 | 라인별 생산량 비교 | 수직 바차트 | **1차 유지** | 라인 간 생산 실적 편차 확인 |
| 차트 | 시간대별 생산/불량 추이 | 복합 차트 (Bar+Line) | **1차 유지** | 생산량 + 불량 추세 결합 표시 |
| 리스트 | 라인별 상태 요약 | 카드형 리스트 | **확인 필요** | `lines.status` 중심 1차 가능, 설비 수준 상세 상태는 제외 필요 |
| 카드 | 전체 라인 OEE | KPI 카드 | **확인 필요** | OEE 산식 확정 및 가동률 계산 근거 추가 필요 |
| 카드 | 예상 종료 생산량 | 예측 수치 카드 | **2차 구현** | `ml_predictions.pred_type = goal` 연계 필요 |
| 카드 | 불량 위험도 요약 | 위험도 카드 | **2차 구현** | 예측 위험도 산출 로직 필요 |
| 차트 | Defect 리스크 게이지 | 게이지 차트 | **2차 구현** | 예측 및 추가 차트 설정 필요 |

### 3.4 생산 현황 보드 (SCR-004)

공용 송출 화면은 원거리 시인성과 단순 정보 전달을 목표로 하므로, 대형 수치 카드와 단순 추세 차트를 중심으로 구성한다. `production_logs`와 `defect_logs` 기반의 대표 수치와 순위형 정보는 1차 구현이 가능하며, 전시 효과 중심의 애니메이션 요소는 2차 구현으로 분리한다.

| 구분 | KPI/요소명 | 시각화 유형 | 구현 분류 | 비고 |
|------|-----------|-----------|---------|------|
| 카드 | 누적 총 생산량 | 대형 KPI 카드 | **1차 유지** | 전시용 대표 수치 |
| 카드 | 당일 양품률 | 대형 KPI 카드 | **1차 유지** | `good_qty` / `inspection_qty` 기반 |
| 차트 | 목표 달성률 | 대형 프로그레스 바 | **1차 유지** | 원거리 시인성 중심 |
| 리스트 | TOP 3 Defect Class | 순위형 리스트 | **1차 유지** | 대표 불량 유형 공유 |
| 차트 | 시간대별 생산 추이 | 심플 라인차트 | **1차 유지** | 단순 흐름 공유 목적 |
| 기타 | 실시간 시계 및 타이틀 | 텍스트 요소 | **1차 유지** | 헤더 영역 기본 요소 |
| 애니메이션 | 전일 대비 품질 개선도 | 증감 애니메이션 카드 | **2차 구현** | 전시 연출 고도화 항목 |

---

## 4. 구현 대상 요약

### 4.1 1차 유지 대상

1차 유지 대상은 `production_logs`, `defect_logs`, `lines`, `product_models`만으로 바로 구현 가능한 항목으로 구성한다. 주요 범주는 다음과 같다:

- 생산량
- 목표 달성률
- 불량률
- 유형별 분포
- 시간대별 추이
- 최근 로그
- 전시용 대표 KPI

이 항목들은 DB 설계상 시간 집계와 합계 조회가 가능하고, 주요 쿼리 패턴에도 직접 연결되므로 9.5.3 구현의 중심 범위로 본다.

### 4.2 확인 필요 대상

확인 필요 대상은 `lines.status`만으로 단순 표시가 가능하더라도, 실제 업무상 의미가 `equipment.status`나 추가 운영 산식과 연결될 수 있는 항목이다.

- 설비 가동 상태
- 라인별 상태 요약
- OEE
- 재검 필요 수량

위 항목들은 구현 전 산식 또는 표시 기준을 한 번 더 정리할 필요가 있다.

### 4.3 2차 구현 대상

2차 구현 대상은 `equipment`, `notifications`, `ml_predictions`와 같이 DB 설계상 2차 구축 범위에 포함된 테이블에 직접 의존하는 항목으로 정의한다.

- 긴급 알람
- 설비 이상 예측
- 예측 불량률
- 예상 종료 생산량
- 위험도 요약
- 신뢰도 분포
- 게이지 차트
- 전시 애니메이션 요소

이 항목은 1차 운영 화면 안정화 이후 고도화 단계에서 구현하는 것이 적절하다.

# 9.5.3.2 공통 UI 규칙 정의

## 핵심 요약

본 문서는 PCB 팩토리 대시보드의 4개 화면에 공통 적용되는 UI 규칙을 정의하여, 카드·차트·테이블·상태 표시 방식의 일관성을 확보하기 위한 기준서이다. 공통 디자인 토큰, 컴포넌트별 시각 규칙, 상태 색상 체계, 로딩 및 오류 표시 원칙을 표준화함으로써 구현 단계에서 화면 간 표현 차이를 최소화하고 유지보수 효율을 높이는 것을 목적으로 한다.

---

## 1. 목적 및 적용 범위

### 1.1 목적

본 문서는 대시보드 내 반복적으로 사용되는 시각 요소의 공통 기준을 정의하여 다음 목적을 달성하고자 한다.

- 화면 간 시각적 일관성 확보
- 사용자 판단 속도 향상
- 프론트엔드 구현 기준 통일
- 후속 차트/테이블 구현 단계의 재작업 최소화

### 1.2 적용 범위

본 규칙은 다음 4개 화면에 공통 적용한다.

- **SCR-001** 현장 대시보드
- **SCR-002** 품질 대시보드
- **SCR-003** 통합 대시보드
- **SCR-004** 생산 현황 보드

> **주의**: SCR-004 생산 현황 보드는 원거리 시인성을 우선하는 전시형 화면이므로, 카드 크기, 글꼴 크기, 애니메이션 범위에 한하여 예외 규칙을 허용한다.

---

## 2. 공통 디자인 토큰

공통 UI 요소의 시각 기준은 다음 디자인 토큰을 기본값으로 사용한다.

### 2.1 색상 및 테두리

| 항목 | 사양 |
|------|------|
| 카드 배경색 | `#0c1520` (var(--bg-card)) |
| 기본 테두리 | `1px solid #1a2840` (var(--border)) |
| 보조 그리드 색상 | `#1e2d40` |
| 라벨 텍스트 색상 | `#4a5d7a` (var(--text-label)) |
| 기본 본문 텍스트 색상 | 밝은 계열 본문색 사용 |
| 강조 성공색 | `#22d3a0` |
| 경고색 | `#fbbf24` |
| 위험색 | `#f87171` |

### 2.2 형태 및 간격

| 항목 | 사양 |
|------|------|
| 카드 모서리 곡률 | `14px` (var(--radius-card)) |
| 카드 내부 여백 | `16px` |
| 카드 간 기본 간격 | `16px` |
| 섹션 간 간격 | `20~24px` |
| 카드 내부 요소 간 간격 | `8px` |

### 2.3 타이포그래피

| 항목 | 사양 |
|------|------|
| 섹션 제목 | 대문자, 10px, letter-spacing 2px |
| 데이터/시간/ID 표기 | DM Mono, monospace |
| 일반 본문 | 시스템 기본 본문 글꼴 또는 프로젝트 표준 글꼴 |
| 보조 라벨 | 작은 크기, 저채도 색상 사용 |

---

## 3. 카드 및 레이아웃 규칙

### 3.1 카드 공통 규칙

카드는 대시보드 정보의 기본 단위로 정의하며, 모든 카드형 컴포넌트는 다음 원칙을 따른다.

- 카드 배경색, 테두리, 모서리 반경은 공통 토큰을 사용한다.
- 카드 내부 여백은 16px로 고정한다.
- 카드 제목은 상단 좌측 정렬을 기본으로 한다.
- 카드 내 주요 수치와 보조 정보는 시각적 위계를 명확히 구분한다.
- 카드 하나에는 하나의 핵심 메시지만 담도록 한다.

### 3.2 카드 제목 규칙

| 항목 | 사양 |
|------|------|
| 정렬 | 좌측 정렬 |
| 글꼴 색상 | `#4a5d7a` |
| 글꼴 크기 | `10px` |
| 자간 | `2px` |
| 텍스트 변환 | Uppercase |
| 제목 하단 여백 | `14px` |

### 3.3 KPI 수치 표기 규칙

KPI 카드는 사용자가 가장 먼저 읽는 영역이므로 수치 표현 방식을 통일한다.

- 주요 수치는 카드 내 최상위 위계로 배치한다.
- 수치 값은 굵은 두께와 큰 글꼴 크기를 사용한다.
- 단위는 값보다 작은 크기로 우측 또는 하단에 배치한다.
- 비율값은 소수점 1자리까지 표기하는 것을 원칙으로 한다.
- 수량 및 건수는 천 단위 구분기호를 적용한다.
- 증감값 또는 보조 설명은 주요 수치 하단에 별도 표기한다.

### 3.4 레이아웃 규칙

- 카드 간 간격은 16px을 기준으로 유지한다.
- 동일 행에 배치되는 카드의 높이는 가능하면 통일한다.
- 중요도가 높은 KPI는 상단 좌측에 우선 배치한다.
- 로그/리스트/테이블은 화면 하단 또는 우측 분석 영역에 배치한다.
- SCR-004는 대형 카드 중심 레이아웃을 우선 적용한다.

---

## 4. 차트 영역 표준

### 4.1 차트 공통 원칙

차트는 추세, 비교, 분포 해석을 지원하는 시각 요소로 정의하며, 가독성과 반복 사용성을 우선한다.

- 차트 컨테이너는 너비 100%, 상대 위치(position: relative)를 유지한다.
- 배경 그리드와 축은 보조 정보 수준으로 표현한다.
- 차트 제목은 카드 제목 규칙을 따른다.
- 범례, 축, 툴팁 형식은 화면마다 달라지지 않도록 공통 옵션을 우선 적용한다.

### 4.2 차트 최소 높이 기준

| 구분 | 최소 높이 |
|------|----------|
| 품질관리자 화면 라인차트 | `210px` |
| 관리자 화면 복합 차트 | `230px` |
| 생산 현황 보드 생산 추이 차트 | `160px` |
| 일반 차트 컨테이너 | 화면 비율에 맞춰 100% 너비 유지 |

### 4.3 차트 시각 규칙

| 항목 | 사양 |
|------|------|
| 기본 글꼴 | DM Mono, monospace |
| 기본 라벨 색상 | `#4a5d7a` |
| 배경 그리드 색상 | `#1e2d40` |
| 축/범례 | 보조 정보 수준으로 약하게 표현 |
| 데이터 색상 | 상태/의미 충돌이 없도록 고정 색상 체계 사용 |

### 4.4 축 및 범례 규칙

- X축은 시간 또는 항목명을 명확히 식별할 수 있도록 축약하되 의미 손실이 없도록 한다.
- Y축 값은 수량, 비율, 건수 등 단위 기준을 일관되게 유지한다.
- 범례는 차트 해석에 반드시 필요한 경우에만 노출한다.
- 범례 위치는 상단 우측 또는 비노출 원칙 중 하나로 통일한다.
- 격자선은 데이터 해석을 방해하지 않도록 약한 대비로 유지한다.

### 4.5 툴팁 규칙

- 툴팁은 항목명: 값 + 단위 형식으로 표기한다.
- 시간축 차트는 시각 정보와 측정값을 함께 표시한다.
- 툴팁 내 숫자 형식은 카드 수치 표기 규칙과 동일하게 적용한다.

---

## 5. 테이블 및 데이터 리스트 규칙

### 5.1 공통 원칙

테이블과 리스트는 세부 정보 확인과 이력 조회를 위한 영역으로 정의한다.

- 헤더와 본문을 명확히 구분한다.
- 숫자, 시간, ID 등은 고정폭 계열 글꼴을 사용한다.
- 스크롤이 필요한 경우 레이아웃이 깨지지 않도록 별도 영역 처리한다.
- 행 단위 정보는 과도한 장식 없이 간결하게 유지한다.

### 5.2 테이블 스타일

| 구분 | 스타일 사양 |
|------|-----------|
| 헤더 (th) | 좌측 정렬, 11px, 600 weight, 하단 테두리 1px |
| 본문 (td) | 12px, 패딩 8px 10px, 하단 테두리 #1a2535 |
| 데이터 폰트 | DM Mono 사용 |
| 상태 셀 | 배지 또는 색상 태그 사용 가능 |
| 스크롤 | 가로 스크롤 및 터치 스크롤 허용 |

### 5.3 리스트 규칙

- 최근 로그 리스트는 최신순 정렬을 기본으로 한다.
- 항목 수가 많은 경우 스크롤 영역을 사용한다.
- 핵심 필드만 우선 노출하고 세부 정보는 확장 또는 상세 테이블로 분리한다.
- 상태값은 색상만이 아니라 텍스트로도 함께 표기한다.

---

## 6. 상태 및 시스템 색상 규칙

### 6.1 장비 및 운영 상태 색상

사용자의 즉각적인 판단을 지원하기 위해 상태 색상은 고정 규칙으로 적용한다.

| 상태 | 텍스트 색상 | 배경색 |
|------|-----------|--------|
| RUN (가동) | `#22d3a0` | `#022c22` |
| IDLE (대기) | `#fbbf24` | `#2c1d00` |
| DOWN (정지) | `#f87171` | `#2d0000` |

### 6.2 경고 및 불량 표시 규칙

| 항목 | 사양 |
|------|------|
| 경고 배너 | 배경 그라데이션 #2d0a00 ~ #3d1500, 테두리 #f87171 |
| NG 태그 | 글꼴 #f87171, 700 weight |
| 주의 상태 | 노란 계열 강조 |
| 정상 상태 | 녹색 계열 강조 |

### 6.3 적용 원칙

- 색상은 의미가 고정된 상태 표현에만 사용한다.
- 동일 의미에 다른 색상을 혼용하지 않는다.
- 색상만으로 상태를 전달하지 않고 텍스트 라벨을 함께 제공한다.

---

## 7. 로딩, 빈 데이터, 오류 상태 규칙

### 7.1 공통 원칙

데이터가 정상적으로 표시되지 않는 상태도 사용자 경험의 일부로 간주하고 표준화한다.

- 로딩, 빈 데이터, 오류 상태는 공통된 구조와 스타일을 사용한다.
- 사용자는 현재 상태를 즉시 이해할 수 있어야 한다.
- 기술적 상세 로그는 사용자 화면에 직접 노출하지 않는다.

### 7.2 로딩 상태

| 항목 | 사양 |
|------|------|
| 레이아웃 | 화면 중앙 정렬 (Flex Center) |
| 높이 | `60vh` |
| 로딩 스피너 | `40px x 40px` |
| 강조색 | `#22d3a0` |
| 애니메이션 | 상단 테두리 회전형 |

### 7.3 빈 데이터 상태

- "데이터 없음" 또는 "집계 기간 내 데이터 없음" 문구를 사용한다.
- 차트는 빈 캔버스 대신 안내 문구 또는 플레이스홀더를 표시한다.
- 테이블은 헤더는 유지하되 본문에 공통 메시지를 표시한다.

### 7.4 오류 상태

- 오류 아이콘, 오류 메시지, 재시도 또는 관리자 확인 안내 문구를 함께 제공한다.
- 오류 아이콘은 32px 수준의 명확한 시각 요소를 사용한다.
- 오류 강조색은 `#f87171`을 사용한다.
- 내부 오류 코드나 개발용 상세 스택 정보는 사용자 화면에 직접 노출하지 않는다.

---

## 8. 반응형 및 화면별 예외 적용 원칙

### 8.1 반응형 원칙

- 테이블은 좁은 화면에서 가로 스크롤을 허용한다.
- 차트는 최소 높이를 유지하되, 화면 폭 축소 시 정보 손실이 크지 않도록 범례와 축 라벨을 조정한다.
- 카드형 영역은 필요 시 다열 구조에서 단일 열 구조로 전환할 수 있다.

### 8.2 화면별 예외 원칙

- **SCR-001**: 즉시 대응 목적이므로 경고, 상태, 최근 로그 영역의 가시성을 우선한다.
- **SCR-002**: 상세 분석 목적이므로 테이블과 추세 차트의 해석성을 우선한다.
- **SCR-003**: 비교 판단 목적이므로 복합 차트와 비교형 차트의 정렬 기준을 엄격히 유지한다.
- **SCR-004**: 전시 목적이므로 대형 수치 카드, 단순 차트, 큰 텍스트 중심으로 구성한다.

# 9.5.3.3 KPI 카드 컴포넌트 구현 메모

## 작업 목적
숫자 중심 핵심 지표를 카드 형태로 구현하여 4개 대시보드 화면(SCR-001~004) 상단 KPI 영역에 재사용 가능한 컴포넌트로 제공한다.

---

## 구현 범위

- 기존 `kpiCardHTML()` 함수 확장 (레거시 `{ label, value, accent, sub }` 호출 호환 유지)
- `renderKpiCards(kpiCards, cols)` 래퍼 함수 추가
- 더미데이터 `MOCK_KPI_CARDS` 배열 + `fetchKpiCards()` 함수 추가
- CSS: 카드 헤더/값-단위 행/증감/상태 배지 스타일 추가

---

## 파일별 반영 내용

| 파일 | 변경 내용 |
|------|----------|
| `dashboard_api.js` | `MOCK_KPI_CARDS` 배열, `fetchKpiCards()` 함수 추가 |
| `dashboard_app.js` | `KPI_STATUS_COLOR/BG`, `KPI_TREND_COLOR/ICON` 상수 추가<br>`kpiCardHTML()` 시그니처 확장 (구 호환 유지)<br>`renderKpiCards()` 함수 추가 |
| `dashboard_style.css` | `.kpi-card-header`, `.kpi-status-badge`, `.kpi-value-row`, `.kpi-unit`, `.kpi-card-footer`, `.kpi-trend` 추가 |

---

## 더미데이터 구조

```js
{
  key:         string,   // 고유 식별자 (data-kpi 속성, API 키 매핑용)
  title:       string,   // 카드 제목
  value:       number,   // 숫자 (정수 → 천단위 포맷, 소수 → 1자리)
  unit:        string,   // 단위 (건, pcs, %)
  helperText:  string,   // 하단 보조 텍스트
  trendType:   'up' | 'down' | 'flat',
  trendValue:  string,   // 증감 표기 (예: "+8", "-2.1", "±0")
  status:      'normal' | 'warning' | 'danger' | 'neutral',
  statusText:  string,   // 상태 배지 텍스트 (예: "급증", "정상", "주의")
}
```

### 4종 더미 예시 매핑

| key | 화면 | 강조 |
|-----|------|------|
| `ng_last_10min` | SCR-001 작업자 | danger (빨강) — 급증 경고 |
| `total_produced` | SCR-001/003/004 | normal (녹색) — 기본 수치 |
| `achievement_rate` | SCR-001/003/004 | warning (노랑) — % 달성률 |
| `oee` | SCR-003 관리자 | warning (노랑) — 상태 뱃지 포함 |

---

## 상태/증감 표기 규칙

### 상태(status) 색상

| status | 텍스트 색 | 배경색 |
|--------|----------|--------|
| normal | `#22d3a0` | `#022c22` |
| warning | `#fbbf24` | `#2c1d00` |
| danger | `#f87171` | `#2d0000` |
| neutral | `#6b7fa3` | `#151f2e` |

### 증감(trendType) 규칙

| trendType | 아이콘 | 색상 | 적용 의미 |
|-----------|--------|------|---------|
| up | ▲ | `#f87171` | 불량/NG는 증가가 위험 → 빨강 |
| down | ▼ | `#22d3a0` | 불량 감소는 긍정 → 녹색 |
| flat | — | `#6b7fa3` | 변화 없음 → 회색 |

> **주의**: `trendType`은 방향만 나타내며, `status`가 의미(좋음/나쁨)를 결정한다. 생산량 증가(`up`)는 `status: normal`, NG 증가(`up`)는 `status: danger`로 구분 설정.

## 구현된 함수

### `resolveTone(value, opts)`

내부 상태값 → `normal / warning / danger / neutral` tone을 반환하는 단일 판정 함수.

- 문자열 값(`RUN`, `IDLE`, `DOWN`, `normal`, `warning`, `danger`)은 고정 매핑
- 숫자 값은 임계치 배열로 분기
- `opts.invert = true` 시 기본 임계치가 반전됨 (자재 잔량 등 낮을수록 danger)
- `opts.thresholds` 로 사용자 정의 임계치 배열 주입 가능

기본 임계치 (일반):

| 범위 | tone |
|------|------|
| 0 ~ 39 | danger |
| 40 ~ 69 | warning |
| 70 ~ 100 | normal |

기본 임계치 (invert):

| 범위 | tone |
|------|------|
| 0 ~ 29 | danger |
| 30 ~ 49 | warning |
| 50 ~ 100 | normal |

---

### `statusBadgeHTML(p)`

재사용 가능한 상태 배지 렌더링 함수.

입력 파라미터:

| 필드 | 타입 | 설명 |
|------|------|------|
| `value` | string | 내부 상태값 (`"RUN"`, `"normal"`, `"danger"` 등) |
| `label` | string | 표시 텍스트 (`"가동중"`, `"정상"`, `"위험"` 등) |
| `tone` | string | (옵션) 직접 tone 지정 시 value 해석 생략 |
| `showDot` | boolean | 점 인디케이터 표시 여부 (기본 `true`) |

지원 상태 매핑 예시:

| value | tone | 표시 텍스트 예시 |
|-------|------|----------------|
| `RUN` | normal | 가동중 |
| `IDLE` | warning | 대기 |
| `DOWN` | danger | 중지 |
| `normal` | normal | 정상 |
| `warning` | warning | 주의 |
| `danger` | danger | 위험 |

접근성: `role="status"`, `aria-label` 적용. 색상과 텍스트를 함께 제공하여 색각 이상자도 인지 가능.

---

### `progressBarHTML(p)`

재사용 가능한 프로그레스 바 렌더링 함수.

입력 파라미터:

| 필드 | 타입 | 설명 |
|------|------|------|
| `label` | string | 항목 라벨 |
| `value` | number | 현재 값 |
| `max` | number | 최대 값 (기본 100) |
| `unit` | string | 단위 (기본 "%") |
| `helperText` | string | 보조 텍스트 |
| `tone` | string | (옵션) 직접 tone 지정 |
| `invert` | boolean | (옵션) 낮을수록 danger |
| `thresholds` | array | (옵션) 사용자 정의 임계치 |

접근성: `role="progressbar"`, `aria-valuenow/min/max`, `aria-label` 적용.

## 적용 화면 및 확장 가이드

### 현재 적용 (1차)

- **SCR-001 현장 대시보드 (`renderWorker`)**: KPI 카드 아래, Defect 바 위에 2열 카드로 배치
  - 좌: 설비 가동 상태 (라인별 배지 + 품질 상태)
  - 우: 목표 달성률 + 자재 잔량 프로그레스 바

### 확장 방법 (2차)

`renderQA()` 품질 상태 배지 추가:
```javascript
// renderQA 내 KPI 카드 데이터에 statusText 필드 추가
{ key: "defect_rate", ..., status: "warning", statusText: "주의" }
// → kpiCardHTML()의 statusBadge 영역에 자동 반영됨
```

`renderManager()` 라인 현황 카드의 `line-status-badge`를 `statusBadgeHTML()`로 교체:
```javascript
// 기존
<span class="line-status-badge" style="color:${clr}; ...">RUN</span>
// 교체
${statusBadgeHTML({ value: l.status, label: l.status === "RUN" ? "가동중" : l.status === "IDLE" ? "대기" : "중지" })}
```

## 구현된 함수

### `renderWorkerLineChart(canvasId, hourly, opts)`

| 파라미터 | 타입 | 설명 |
|---------|------|------|
| `canvasId` | string | canvas 요소 ID |
| `hourly` | Array | `{ time, produced, defect_rate, predicted_defect }` 배열 |
| `opts.showDefect` | boolean | 불량률 라인 표시 여부 (기본 `true`) |

#### 시리즈 구성

| 시리즈 | Y축 | 색상 | 비고 |
|--------|-----|------|------|
| 생산량 (pcs) | 좌측 (y) | `#22d3a0` | fill: true, 면적 강조 |
| 불량률 (%) | 우측 (y2) | `#fbbf24` | `opts.showDefect: false` 시 비표시 |

#### 축 규칙

- **X축**: `time` 문자열 배열. 항목 10개 초과 시 `skipStep` 계산으로 라벨 간격 자동 조정 (겹침 방지)
- **Y축(생산량)**: 1000 이상 → `1.2k` 포맷 자동 적용
- **Y2축(불량률)**: 단위 `%` 자동 suffix, `suggestedMax: 6`

#### 툴팁
```
시간: 10:00
 생산량: 1,450 pcs
 불량률: 1.9%
```

- `interaction: { mode: "index", intersect: false }` — hover 시 같은 X 위치의 모든 시리즈 동시 표시
- 배경 `#0a111cee`, 테두리 `#1e2d40` — 기존 다크 테마와 일치

## 더미데이터 연결

`MOCK_HOURLY` (api.js 기존 상수)를 그대로 사용. 별도 상수 추가 없음.
```js
// MOCK_HOURLY 구조 (기존)
```

- `defect_rate: null` 구간은 `spanGaps: false`로 라인 끊김 처리 (예측 구간 분리)

## 충돌 검토

| 항목 | 결과 |
|------|------|
| `chartInstances` Map | 공유 사용, `destroyChart()` 자동 호출됨 — 충돌 없음 |
| canvas ID 중복 | `qaBarChart`, `managerLineBarChart` 신규 ID — 충돌 없음 |
| `renderManager()` lineCards 제거 | 동일 데이터를 차트로 대체. 상태/불량률 상세는 게이지 섹션에서 확인 가능 |
| CSS 변경 | 없음. 기존 `.chart-wrap`, `.card`, `.section-title` 재사용 |

# 9.5.3.7 도넛차트 구현 메모

## 1. 구현 개요

| 항목 | 내용 |
|------|------|
| 구현 방식 | Chart.js `type: "doughnut"` + 인라인 플러그인 |
| 핵심 함수 | `renderDonutChart(canvasId, items, opts)` |
| 보조 함수 | `donutLegendHTML(items, total)`, `buildStatusDistribution(lines)` |
| 적용 화면 | Worker(검사 결과 비율), Manager(라인 가동 현황) |
| QA 화면 | 기존 `renderDefectPieChart` 유지 (변경 없음) |

---

## 2. 수정 파일 목록

| 파일 | 변경 내용 |
|------|----------|
| `dashboard_charts.js` | `renderDonutChart()` 파일 끝에 append |
| `dashboard_api.js` | `MOCK_INSPECTION_RESULT` 상수 추가, `fetchAll` 반환값에 `inspectionResult` 추가 |
| `dashboard_app.js` | `donutLegendHTML()`, `buildStatusDistribution()` 추가; `renderWorker`/`renderManager` 템플릿에 canvas 삽입; `activateTab` 차트 호출 추가 |

---

## 3. 함수 인터페이스

### renderDonutChart(canvasId, items, opts)

```js
// items 필수 구조
[{ label: "양품", value: 7840, color: "#22d3a0" }]

// opts 선택 항목
{
  centerText:    "93.2%",   // 중심 큰 텍스트 (null/undefined → 숨김)
  centerSubText: "양품률",  // 중심 하단 보조 텍스트
  maxItems:      6,         // 초과 시 "기타" 합산 (기본 6)
  cutout:        "62%",     // 도넛 두께 (기본 "62%")
}
```

### donutLegendHTML(items, total)

- `total > 0` → 비율(%) 표시 / `total === 0` → 값 표시
- 반환값: HTML 문자열 → `element.innerHTML`에 삽입
- `esc()` 의존으로 `dashboard_app.js`에 위치

### buildStatusDistribution(lines)

- MOCK_LINES 배열에서 status별 카운트 집계
- 반환 형태: `[{ label, value, color }]`
- lines API 연동 시 자동 반영

---

## 4. 적용 위치 및 데이터 흐름

| 화면 | canvasId | 범례 id | 데이터 소스 | 중심 수치 |
|------|----------|---------|------------|----------|
| Worker | `workerInspectionDonut` | `workerInspectionLegend` | `cache.inspectionResult` | 양품률 % |
| Manager | `managerStatusDonut` | `managerStatusLegend` | `buildStatusDistribution(lines)` | 전체 라인 수 |
| QA | `qaPieChart` | HTML 범례 (기존) | `cache.defects` | 없음 |

---

## 5. 항목 수 많을 때 처리 규칙

| 조건 | 처리 방식 |
|------|----------|
| `items.length ≤ maxItems` (기본 6) | 전체 표시 |
| `items.length > maxItems` | 상위 N개 + "기타" 합산 (`color: "#3a4a5e"`) |
| 범례 | 원본 items 전체 표시 (차트와 별도) |
| 상세 값 | tooltip에서 건수 + 비율 제공 |

현재 적용 데이터 항목 수:
- 검사 결과: 3개 → 기타 처리 불필요
- 라인 상태: 3~4개 → 기타 처리 불필요

항목이 7개 이상인 경우 옵션으로 제어:
```js
renderDonutChart("myCanvas", items, { maxItems: 5 });
```

---

## 6. 중심 수치 표시 규칙

| 상황 | centerText 권장값 |
|------|-----------------|
| 비율 강조 (검사 결과) | `"93.2%"` (양품률) |
| 총합 강조 (라인 현황) | `"4"` (전체 라인 수) |
| 표시 불필요 | `opts.centerText` 생략 또는 `null` |

---

## 7. API 연동 전환 방법

```js
// dashboard_api.js: MOCK → fetch 교체
async function fetchInspectionResult() {
  return await apiFetch("/api/v1/dashboard/inspection-result");
}

// fetchAll에서
inspectionResult: await fetchInspectionResult(),
```

`buildStatusDistribution(lines)`은 lines API 연동 후 자동 반영.

---

## 8. 알려진 제약

- `donutLegendHTML`은 `esc()`에 의존 → `dashboard_charts.js`가 아닌 `dashboard_app.js`에 위치
- Chart.js 인라인 플러그인 배열은 Chart.js 4.x 기준 동작
- `cutout` 80% 이상이면 중심 텍스트 공간 부족 → 기본값 `"62%"` 유지 권장

# 9.5.3.8 로그리스트 구현 메모

## 작업 목적

- 최근 이벤트, 알람, 검출 기록을 공통 구조의 리스트로 표시한다.
- 기존 실시간 NG 로그 카드는 유지하고, 신규 로그 리스트 카드를 추가한다.

## 구현 내용

- `recentLogs` 더미 데이터를 `dashboard_api.js`에 추가했다.
- Worker 화면 하단에 `최근 이벤트 / 알람 로그` 카드를 신규 추가했다.
- 로그 정렬, 상태 매핑, 렌더를 위한 공용 함수 3개를 `dashboard_app.js`에 추가했다.
- 빈 데이터 문구, 최신순 정렬, 최대 높이/내부 스크롤, 메시지 말줄임을 적용했다.

## 적용 파일

- `nginx/html/js/wed_dashboard/dashboard_api.js`
- `nginx/html/js/wed_dashboard/dashboard_app.js`
- `nginx/html/css/wed_dashboard/dashboard_style.css`

## 로그 데이터 구조

```js
{
  time: "14:59:21",
  line: "LINE B",
  status: "NG",
  message: "short 불량 4건 연속 검출, 비전 카메라 점검 필요",
  type: "defect",
  priority: "high",
  rawTimestamp: "2026-03-14T14:59:21+09:00"
}
```

## 정렬 방식

- 렌더 직전에 `sortRecentLogs(logs)`를 호출한다.
- 기준 필드는 `rawTimestamp`이며, 값이 비정상인 경우 `time` 문자열을 fallback으로 사용한다.
- 최신순(내림차순)으로 정렬 후 `limit` 개수만 화면에 표시한다.

## UI/스타일 처리

- 카드/섹션은 기존 `card`, `section-title` 스타일을 재사용했다.
- 리스트 컬럼은 `시간 / 라인 / 상태 / 메시지` 4열 그리드로 구성했다.
- `.log-list-body`에 `max-height: 260px`, `overflow-y: auto`를 적용했다.
- 메시지는 `ellipsis` 처리해 가독성을 유지했다.
- 상태 표시는 기존 `statusBadgeHTML()`과 `status-badge` 테마를 재사용했다.

## 더미 데이터 연결 방식

- `fetchAll()` 반환에 `recentLogs`를 추가했다.
- `activateTab()`에서 `cache.recentLogs`를 가져와 Worker 렌더에 전달한다.
- Worker 화면에서 `renderRecentLogList(recentLogs, { limit: 12 })`로 렌더한다.

## 확인 필요 사항

- `recentLogs`의 `type`, `priority`를 UI에 직접 노출할지 여부
- 실제 API 연동 시 `rawTimestamp` 포맷(타임존 포함 ISO8601) 고정 여부
- 메시지가 긴 다국어 데이터일 때 줄바꿈/툴팁 정책 고도화 필요 여부

## 다음 연계 Task

- `recentLogs`를 백엔드 API 응답으로 교체
- 로그 타입/우선순위 필터(알람만, 검출만) 추가
- 클릭 시 상세 로그 패널(모달 또는 우측 패널) 연계

# 9.5.3.9 상세 테이블 구현 메모

## 작업 목적

- 차트 영역에서 다 담지 못하는 상세 데이터를 하단 카드형 테이블로 조회한다.
- 기존 로그 리스트와 동일한 설계 계열을 유지하되, 이벤트 스트림과 조회형 레코드의 역할은 분리한다.

## 1차 확정 범위

- QA 화면: `검사 결과 목록`
- Manager 화면: `라인별 생산 현황`

Worker 화면은 기존 로그 리스트를 유지하며, 이번 단계에서 상세 테이블을 추가하지 않는다.

## 적용 파일

- `nginx/html/js/wed_dashboard/dashboard_app.js`
- `nginx/html/js/wed_dashboard/dashboard_api.js`
- `nginx/html/css/wed_dashboard/dashboard_style.css`

## 컬럼 정의

### QA 화면: 검사 결과 목록

- 검사시각
- 라인명
- 품목명
- 불량유형
- 판정결과
- 검사자

### Manager 화면: 라인별 생산 현황

- 라인명
- 목표수량
- 생산수량
- 달성률
- 불량률
- 상태
- 갱신시각

## 정렬 기준

- 시간성 데이터는 시간 역순 정렬
- Manager의 라인별 생산 현황 표는 라인명 오름차순 정렬

## 공통화 범위

- 공통 상세 테이블 렌더 함수
- 기본 정렬 함수
- 셀 타입별 포맷 함수
- 상태값 배지 매핑 처리

화면별 차이는 컬럼 메타데이터와 데이터 세트로만 분리한다.
공통 렌더 함수 내부의 화면별 분기 로직은 최소화한다.

## 표시 규칙

- 헤더 고정은 1차 구현에서 적용하지 않는다.
- 카드 내부에만 세로 스크롤을 적용한다.
- 테이블 최대 높이는 `360px`로 고정한다.
- 긴 텍스트는 1줄 ellipsis와 `title` 속성만 적용한다.
- 별도 tooltip 컴포넌트는 사용하지 않는다.
- 숫자 셀은 우측 정렬한다.
- 시간 및 ID 계열은 `DM Mono` 기반 모노스페이스 스타일을 적용한다.
- 상태 셀은 기존 `statusBadgeHTML()`을 재사용한다.

## 더미데이터 기준

- 실제 API와 동일한 키 이름을 사용한다.
- 시간값은 ISO8601 원시값으로 유지한다.
- 상태값은 enum으로 유지한다.
- `null` 가능 필드는 더미에서도 그대로 유지한다.
- 숫자 필드는 문자열이 아닌 숫자 타입으로 유지한다.

## 역할 분리 원칙

- 로그 리스트: 최신 이벤트 모니터링
- 상세 테이블: 필드 비교와 조회 중심의 레코드 목록

같은 하단 상세 영역 계열이지만 동일 컴포넌트로 합치지 않는다.

## 후속 연계 시 주의사항

- 실제 API 연동 시 시간 필드 포맷은 ISO8601 기준으로 유지한다.
- 상태 enum 추가 시 배지 매핑 사전을 먼저 갱신한다.
- 화면별 컬럼 변경은 공통 렌더 함수가 아니라 컬럼 메타데이터에서 처리한다.
- 필터, 페이징, 드릴다운은 이번 단계 범위에 포함하지 않는다.

# 9.5.3.10 차트 공통 유틸 정리

- 로그/테이블 긴 텍스트 ellipsis 유지
- 내부 스크롤/가로 스크롤 동작 유지
- 비대상 화면/기능 회귀 없음

## 10. 추후 개선 후보
- formatter 모듈 분리(전역 함수 충돌 방지)
- 도넛 범례 인라인 스타일을 CSS 클래스로 이동
- 축 helper를 차트 타입별 preset으로 세분화
- 결측/0/NaN 표시 정책을 화면 단위로 더 명확히 분리
- QA KPI를 raw number 기반으로 점진 전환

## 11. 주의사항
- `dashboard_app.js`는 전체 함수 재정의가 아니라 필요한 구간만 부분 치환해야 한다.
- 도넛 데이터는 차트 렌더 전/내부에서 중복 가공하지 않도록 주의한다.
- `formatChartNumber/formatChartPercent` 결측 정책(`-`)을 임의 변경하면 기존 UI 해석이 달라질 수 있다.
- `compactK`는 축 전용 규칙이다. KPI/툴팁/테이블로 확장하지 않는다.
- 공통화 범위는 “옵션/포맷/헬퍼”까지로 제한하고, 화면별 비즈니스 의미는 renderer에 남긴다.
